# Phase 4 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Carry-Forward Note

This phase introduces **no new visual identity** — it reuses the site-wide design system locked in `02-UI-SPEC.md` (Dawn Pink / Woodsmoke / Wild Strawberry, Delight font, 4-role/2-weight typography, 8-point spacing scale) and the "single flowing page, no boxed panels, no portrait" layout pattern established for the About page (`03-CONTEXT.md` D-02/D-03, `03-UI-SPEC.md` Layout Notes). Nothing in `04-CONTEXT.md` requests a new color, font, or spacing scale. This document:

1. Confirms the existing tokens apply unchanged (no re-litigation).
2. Declares role assignments, page structure, and copywriting specific to the two new legal pages this phase adds (mentions légales, privacy policy — the cookie/consent disclosure is a **section inside** the privacy policy page, not a third page or a banner, per `04-CONTEXT.md` D-01/D-02).
3. Resolves the two footer-wording/ordering decisions `04-CONTEXT.md` explicitly left to Claude's Discretion.
4. Explicitly does **not** attempt to resolve the two content-level research gates flagged in `04-CONTEXT.md` (D-03: CNIL guidance on the locale cookie; D-08: LCEN Article 6-III identity-disclosure requirement). Those are legal-research questions for `RESEARCH.md`/the planner, not visual/interaction contract questions — this document only fixes how whatever text they produce gets *displayed*.

Sources used to pre-populate this contract: `04-CONTEXT.md` `## Decisions` and `## Claude's Discretion` (all locked or explicitly delegated), `src/layouts/BaseLayout.astro` (existing tokens, existing footer markup), `03-UI-SPEC.md` (established role/layout precedent for a single informational page), `src/pages/about.astro` (concrete implementation reference for the page-wrapper/typography pattern). No `RESEARCH.md` exists yet for this phase (runs during `/gsd:plan-phase`, after this document) — the two research gates above do not change any visual/interaction contract below, only the final legal copy that gets poured into the structure this contract fixes.

## Spacing Scale

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding |
| sm | 8px | Compact element spacing (footer link internal padding) |
| md | 16px | Default element spacing (subheading-to-paragraph gap, page horizontal padding, footer text-to-links-row gap) |

| 2xl | 48px | Major section breaks (between legal-page sections, e.g. "Éditeur" → "Hébergement" → "Données personnelles" → "Cookies") |

- **44px minimum tap target** does **not** apply to the two new footer links — they are inline text links inside a `<nav>`, rendered identically to the existing `.nav-link` header pattern, which itself carries no `min-height` override. Consistent with existing precedent; not a new exception.
- No other exceptions. This phase introduces no forms, no buttons, no interactive controls beyond ordinary hyperlinks.

## Typography

Unchanged 4-role/2-weight system from Phase 2 — reused, not redeclared:

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Label | 14px | 300 (Light) | 1.5 |
| Body | 16px | 300 (Light) | 1.5 |
| Heading | 20px | 900 (Black) | 1.2 |
| Display | `clamp(2.5rem, 12cqi, 6.5rem)` (fluid, container-query based — see `02-UI-SPEC.md` Typography for the full mechanism) | 900 (Black) | 1.0 |

Exactly 4 roles, exactly 2 weights (300, 900) — no new role or weight introduced this phase. Both legal pages' subsections (e.g. Privacy Policy's "What we collect" / "Why" / "How long") reuse the **Heading** role regardless of semantic HTML nesting depth (h2 vs h3) — the visual system has no third heading size, so nested subsections stay visually identical to top-level sections and rely on document order/semantic tags (not size) to convey hierarchy.

## Layout Notes (Phase 4 specifics, not covered by the template's fixed sections)

### Shared Page Pattern (both new pages)

- Single page, flowing sections top-to-bottom — no tabs, no sub-navigation, no visually-boxed panels between sections (reuses `03-CONTEXT.md` D-02's "no boxed panels" precedent, which `04-CONTEXT.md`'s Claude's Discretion note explicitly asks to replicate).
- Max content width: `640px`, centered — identical reading-width precedent as About/Contact/homepage (`.about-page`/`.contact-page` wrapper pattern in the existing codebase). Reuse the exact CSS: `max-width: 640px; margin: 0 auto; padding: var(--space-2xl) var(--space-md);`.
- No portrait/photo, no hero imagery on either page — text-only, consistent with the informational nature of legal content and the site-wide "no portrait" precedent.
- Vertical rhythm: `var(--space-2xl)` (48px) between major sections (e.g. "Éditeur du site" → "Hébergement" → "Statut" on mentions légales; each data-flow subsection on the privacy page); `var(--space-md)` (16px) between a subheading and its immediately-following paragraph — same rhythm as the About page's bio-to-atelier break.
- Both pages get real `<h1>` titles per locale (not a shared generic "Legal" heading) — mirrors the About/Contact pattern of locale-specific, translated page titles.

### Mentions Légales Page — Section Order

1. `<h1>` — "Mentions légales" / "Legal notice" (Display role, plain, no panel).
2. "Éditeur du site" / "Site publisher" (Heading role) — identity disclosure content; exact wording pending the D-08 LCEN research gate, rendered as normal (non-italic) Body text once settled.
3. "Hébergement" / "Hosting" (Heading role) — names OVH per `04-CONTEXT.md` D-07 (describes the confirmed Phase-5 end state, not today's GitHub Pages staging reality).
4. "Statut" / "Status" (Heading role) — business-status line, italic-placeholder treatment (see Typography/Copywriting above) until SIRET is confirmed.

### Privacy Policy Page — Section Order

1. `<h1>` — "Politique de confidentialité" / "Privacy Policy" (Display role, plain, no panel).
2. Optional one-line intro (Body role) framing the page as a lightweight, plain-language notice (per `04-CONTEXT.md` D-04 — not a formal Article 13 processor list).
3. "Formulaire de contact" / "Contact form" (Heading role) — describes the direct-send-to-Romane's-own-OVH-mailbox flow (`04-CONTEXT.md` D-05), explicitly framed as no third-party data processor involved.
4. "Contenu du site (Sanity)" / "Site content (Sanity)" (Heading role) — one-line disclosure that gallery/site content is fetched at build time only; no visitor data reaches Sanity.
5. "Journaux d'hébergement" / "Hosting logs" (Heading role) — brief disclosure that the web host automatically logs visitor IP/user-agent.
6. "Cookies" (Heading role) — folds in the locale-preference cookie disclosure (`04-CONTEXT.md` D-01): names the `ajs_locale` cookie, states it is strictly functional/exempt, and explains there is no accept/reject banner because there is no tracking/analytics cookie anywhere on the site. This section is the **sole** place cookie information appears — confirms `04-CONTEXT.md` D-02's decision that no separate cookie-banner UI or dedicated cookie-policy page exists.

### Footer (site-wide change, `src/layouts/BaseLayout.astro`)

- Current footer (`<footer class="chrome-band"><p class="footer-text">…</p></footer>`) gains a second row: a `<nav>` containing the two new legal-page links, rendered on **every page** (not scoped to About/Contact), per `04-CONTEXT.md` D-06.
- Layout: flex column on mobile (`gap: var(--space-md)`, links row below the footer text), flex row on desktop (`≥768px`, same breakpoint the header already uses) with the footer text on the left and the links `<nav>` on the right, `gap: var(--space-xl)` between the two links themselves — mirrors the header's existing `.site-nav` gap value exactly.
- Link styling: identical to the existing `.nav-link` class (14px/300 Label role, `text-decoration: none`, underline on hover/focus-visible, standard site-wide `:focus-visible` double-ring reused unchanged). No new visual treatment invented — this is a straight reuse of an existing pattern, applied to a new location.
- `aria-label="Legal"` (or locale-equivalent, e.g. `"Informations légales"`) on the new footer `<nav>` to distinguish it from the header's `aria-label="Primary"` nav for assistive-tech users.
