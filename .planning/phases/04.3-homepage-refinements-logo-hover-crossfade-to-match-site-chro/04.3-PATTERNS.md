## Pattern Assignments

### `src/components/HomeCarousel.astro` — `.home-logo` hover/crossfade (D-01/D-02)

**Analog:** `src/layouts/BaseLayout.astro` lines 109-113 (markup) and 313-367 (CSS)

**Current HomeCarousel markup to replace** (lines 81-84):
```astro
<a href={homeHref} class="home-logo" aria-label={wordmark}>
  <img src={logoWhiteSrc} alt="" class="home-logo__img home-logo__img--white" />
  <img src={logoBlackSrc} alt="" class="home-logo__img home-logo__img--black" />
</a>
```

**BaseLayout's markup to mirror** (lines 109-113):
```astro
<a href={homeHref} class="logo-mark" aria-label={siteTitle}>
  <span class="logo-mark__chip" aria-hidden="true"></span>
  <img src={logoBlackSrc} alt="" class="logo-mark__img logo-mark__img--default" />
  <img src={logoWhiteSrc} alt="" class="logo-mark__img logo-mark__img--hover" />
</a>
```
Note BaseLayout's convention: `--default` = black (shown normally), `--hover` = white (fades in via `position:absolute; opacity 0→1`). HomeCarousel currently names its two `<img>`s by color (`--white`/`--black`) rather than by role (`--default`/`--hover`); when reworking, keep whichever naming style is chosen consistent with the new mode-conditional CSS below — the key structural difference from the old markup is adding the `.logo-mark__chip` span and switching from `display:none/block` per-mode swapping to the opacity-crossfade mechanism.

**BaseLayout's CSS to mirror** (lines 313-367):
```css
.logo-mark {
  position: relative;
  display: inline-flex;
  align-items: center;
  text-decoration: none;
}

.logo-mark__chip {
  position: absolute;
  inset: calc(-1 * var(--space-sm));
  background-color: var(--color-ink);
  border-radius: var(--radius-none);
  opacity: 0;
  transition: opacity 150ms ease;
  z-index: 0;
}

.logo-mark__img {
  position: relative;
  z-index: 1;
  display: block;
  width: 56px;
  height: auto;
}

.logo-mark__img--hover {
  position: absolute;
  inset: 0;
  opacity: 0;
  transition: opacity 150ms ease;
}

/* Solid header: default = black logo; hover/focus-within cross-fades to
   the white variant over the dark chip. */
.site-header--solid .logo-mark:hover .logo-mark__chip,
.site-header--solid .logo-mark:focus-within .logo-mark__chip,
.site-header--solid .logo-mark:hover .logo-mark__img--hover,
.site-header--solid .logo-mark:focus-within .logo-mark__img--hover {
  opacity: 1;
}

/* Transparent header: white logo variant shown by default (over the dark
   scrim already provided by the header background) — no hover swap. */
.site-header--transparent .logo-mark__img--default {
  opacity: 0;
}

.site-header--transparent .logo-mark__img--hover {
  opacity: 1;
}
```

**How to adapt for HomeCarousel's two modes (D-02):** BaseLayout scopes hover behavior by `.site-header--solid` vs `.site-header--transparent` (a class on the `<header>` set via the `headerVariant` prop). HomeCarousel already has an equivalent mode selector on its root: `.home[data-display-mode='carousel']` / `.home[data-display-mode='grid']` (see existing selectors at lines 308, 321, 338, 342, 359-367 for the established convention of scoping rules to `.home[data-display-mode='...']`). Grid mode should get the BaseLayout `--solid` hover rule (`.home[data-display-mode='grid'] .home-logo:hover .home-logo__chip, ...`); carousel mode should get the BaseLayout `--transparent` static rule (white shown by default via `.home[data-display-mode='carousel'] .home-logo__img--hover { opacity: 1; }` and `--default { opacity: 0; }`, no `:hover`/`:focus-within` rule at all — mirroring that `--transparent` intentionally omits hover rules).

**Analog:** own toggle-button click handlers (lines 263-274) — the CTA should call the same `showGrid()` + `setActiveToggle()` logic, not `location.href`.

```astro

```typescript
root.querySelectorAll<HTMLButtonElement>('[data-action="show-grid"]').forEach((btn) => {

Delete `ctaHref` const (line 62) since `galleriesHref` it referenced is also being deleted (D-04).

**`render()` must set both `textContent` and `href` on every call** (current, lines 194-213, `titleEl` is currently an `<h2>` — must become/wrap an `<a>`):
```typescript

...

  ...

    titleEl.textContent = gallery.title.toUpperCase();
  }
  ...

```
Two viable adaptations, both requiring `getRelativeLocaleUrl` to be usable client-side (it isn't — it's an Astro/server import) or the href to be precomputed server-side per gallery and looked up by index:
1. **Precompute hrefs server-side into the data island** (recommended, consistent with existing "thread it through the data island" convention noted in CONTEXT.md code_context): add `data-href={getRelativeLocaleUrl(locale, \`galleries/${gallery.slug}\`)}` directly to each `<li>` (alongside or instead of raw `data-slug`) so the client script just reads `gallery.href` and does `titleEl.setAttribute('href', gallery.href)` — avoids reimplementing locale-prefix logic in plain JS.
2. Reconstruct the href client-side from `slug` + a locale prefix already known to the script (less clean, would need the FR/EN base path threaded in too) — not preferred; option 1 matches the "no extra plumbing beyond what's already in the data island" pattern.

Markup change: swap `<h2 class="home-hero__title" data-role="gallery-title">` for either an `<a>` styled to look like the heading (simplest, keeps the DOM flat, still update `render()` to touch `.textContent`/`href` on the same element) or keep the `<h2>` and nest an inner `<a data-role="gallery-title">` (heading semantics preserved, link target more precise) — CONTEXT.md doesn't lock this choice; either satisfies "the hero caption title... must become a link."
