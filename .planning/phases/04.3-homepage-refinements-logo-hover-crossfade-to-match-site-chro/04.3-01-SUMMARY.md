# Phase 04.3 Plan 01: HomeCarousel Refinements Summary

**Homepage logo now crossfades like BaseLayout's chrome, the accent-panel CTA switches display mode instead of dead-linking to /galleries, the carousel/grid toggle is icon-based, the wordmark reads on three lines, the carousel hero title is a synced live link, and a genuine mobile stacking bug (accent panel hidden behind the hero photo, not an image collapse) is fixed with a permanent Playwright regression guard.**

## Accomplishments
- `.home-logo` gained BaseLayout's exact chip+two-stacked-`<img>` opacity-crossfade mechanism, scoped by `data-display-mode` (grid mirrors `--solid`'s hover/focus crossfade; carousel mirrors `--transparent`'s always-white, no-hover state) — verified live via computed-style checks in both modes/states.
- Homepage nav (`.home-nav`) reduced to exactly 3 links (Home/About/Contact); `galleriesLabel`/`galleriesHref`/`ctaHref` consts removed.
- Accent-panel CTA converted from a dead `<a href={ctaHref}>` to `<button data-action="show-grid">`, reusing the existing toggle-button wiring; client script updated to explicitly target the real Grid/Carousel toggle buttons for `is-active` state so the CTA (which shares the same `data-action`) doesn't steal it.
- Carousel/grid toggle buttons now render inline dependency-free SVG icons with `aria-label` carrying the locked Carrousel/Grille (Carousel/Grid) strings — verified all pre-existing `getByRole('button', { name: ... })` assertions still resolve.
- Wordmark renders as three explicit `<br />`-separated lines at both `.home-hero__wordmark` and `.home-grid__wordmark`; the flat `wordmark` const is unchanged and still supplies the logo's `aria-label`.
- Carousel hero title (`[data-role="gallery-title"]`) is now an `<a>` with a build-time-precomputed `data-href` threaded through the data island exactly like `title`/`heroSrc`/`alt`; `render()` calls `setAttribute('href', ...)` alongside the existing `textContent` update so the link target stays in sync on every prev/next/auto-advance.
- Root-caused and fixed the real mobile "hero disappears" bug: on mobile, `.home-hero__accent` switching to `position: static` silently dropped its `z-index: 3` out of effect (z-index has no effect on statically-positioned boxes), so the opaque hero photo (a later-painted, position:absolute layer) visually covered the accent panel's wordmark/intro/CTA content completely, even though every element individually reported non-zero size and `visibility: visible`. Fixed by switching to `position: relative` (keeps z-index active, stays in-flow) and neutralizing the inherited `right`/`bottom` offsets that `position: relative` (unlike `static`) would otherwise honor. Also bumped the mobile prev/next arrows to `z-index: 4` since restoring the accent panel's correct stacking exposed a previously-invisible overlap with the arrows against the now-taller three-line-wordmark accent panel.
- Added a permanent 375x667 Playwright regression test asserting `.home-hero` height > 300px, the hero image is visible with height > 0, and — going beyond the plan's literal acceptance criteria to actually guard the bug found — that the accent CTA is clickable (not obscured) and triggers grid mode.

## Decisions Made
- Root-caused D-08 via live Chromium measurement (elementFromPoint, computed styles, and screenshots with a controlled opaque test image to isolate the CSS bug from unrelated sandbox network flakiness against the real Sanity CDN) as an accent-panel z-index/stacking issue, not an image-height-collapse issue as the plan's three investigation candidates speculated. The image's `height: 100%` was confirmed to correctly resolve against the mobile `min-height: 600px` box in Chromium at every test I ran — the actual bug was the opaque hero photo (position:absolute, painted late) visually covering the accent panel (which had become position:static, and therefore lost its z-index) underneath it.
- Bumped `.home-hero__arrow` z-index from 3 to 4 (not part of the plan's explicit fix list) because correctly restoring the accent panel's visibility exposed a previously-hidden overlap between the vertically-centered arrows and the now-taller (three-line wordmark, Task 1) accent panel on mobile — classified as Rule 1 (auto-fix bug: the arrows were unclickable/invisible in that overlap region) rather than a separate deviation requiring sign-off, since it's a one-line, low-risk fix directly required to make the mobile hero fully usable (not just visually present).

## Issues Encountered

- **Sandbox build environment bug (not a code issue, no source fix applied):** `npm run build` and `npx vitest run` both fail inside this worktree (`.claude/worktrees/agent-ad3acc0720b862daf/`, nested under the main repo) with `Tsconfig not found astro/tsconfigs/strict` from rolldown-vite's native tsconfig auto-discovery. Root-caused via elimination (temporarily removing/renaming the worktree's own `tsconfig.json` had zero effect on the error) as rolldown's directory-walk-up tsconfig/root discovery treating the worktree's `.git` *file* (a worktree pointer, not a real `.git` directory) as "not a real root," continuing to walk up to the main repo's `.git` *directory* and using its (unedited) `tsconfig.json`/lack of `node_modules` instead of the worktree's own. Confirmed by building/testing from an isolated scratchpad copy (source files copied out, `node_modules` symlinked back to the worktree) where the identical `tsconfig.json` builds and tests cleanly — proving this is purely an artifact of the nested-worktree-inside-repo sandbox layout, not a real project bug. All verification for this plan (`npm run build`, `npx playwright test homepage.spec.ts`, `npx vitest run`) was run against that isolated scratchpad copy instead, using the sandbox's existing `SANITY_PROJECT_ID`/`SANITY_DATASET`/`SANITY_API_READ_TOKEN` env vars (trimmed of an unrelated leading-whitespace artifact in how they were injected into this shell, without printing their values). No files outside `src/components/HomeCarousel.astro` and `tests/e2e/homepage.spec.ts` were changed as a result of this investigation — a speculative `tsconfig.json` edit was tried, found to have no effect, and fully reverted (confirmed via `git status`/`git diff` showing zero diff on `tsconfig.json` before the final commits).
- Real Sanity CDN network requests were intermittently flaky in this sandbox (`net::ERR_CONNECTION_RESET`/`ERR_ABORTED` on some image fetches, and one borderline ~25-30s page-load timeout on the i18n switcher test) — unrelated to any code change in this plan.
