<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner → OVH host | Deploy transfers build output using stored credentials |
| Sanity webhook → GitHub REST API | External service triggers CI using a PAT |
| GitHub Secrets → workflow env | Secrets are injected into the build/deploy job |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-FTP | Information Disclosure | OVH deploy transport + credentials | mitigate | `protocol: ftps` (encrypted), never plain `ftp`; host/user/password only in GitHub Secrets, referenced via `secrets.*`, never echoed. |
| T-01-PAT | Elevation of Privilege | GitHub PAT used by Sanity webhook | mitigate | Fine-grained PAT scoped to this one repo, `Actions: write` + `Contents: read` only; stored only in Sanity's webhook config, never in the repo (RESEARCH.md line 482). |
| T-01-SEC | Information Disclosure | secrets in CI logs | mitigate | Use `secrets.*` context (auto-masked by Actions); no `echo`/`env` dumps of secret values; `git grep` gate for committed secrets in Task 2. |
| T-01-SC | Tampering | third-party Actions (setup-node, FTP-Deploy-Action) | mitigate | Pin FTP-Deploy-Action to an exact tag (`@v4.4.0`); use official `actions/*` at pinned major; reject git-clone-based OVH actions (Pitfall 2). |
| T-01-SPOOF | Spoofing | repository_dispatch trigger | accept | The dispatch endpoint requires the fine-grained PAT to invoke; worst case of a spurious trigger is an extra idempotent rebuild-and-deploy of already-published content (no data mutation). Low risk. |
</threat_model>
