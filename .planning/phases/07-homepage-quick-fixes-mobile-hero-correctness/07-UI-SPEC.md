## Spacing Scale

Existing site-wide tokens (`src/layouts/BaseLayout.astro` `:root`) — reuse as-is, no new tokens needed for this phase:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding |
| sm | 8px | Compact element spacing |
| md | 16px | Default element spacing |
| lg | 24px | Section padding |
| xl | 32px | Layout gaps |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing |

Exceptions for this phase:

- **`--tap-target-min` (44px)** — already an established token (`HomeCarousel.astro:727`, `LanguageSwitcher.astro:40`). Reused, not new. HOME-05's toggle fix and HOME-04's Instagram nav link both must keep their *clickable/tappable* box at or above this floor even though their *visible* box is smaller (see Layout & Interaction Notes below).
- **Toggle visible square box: `28px × 28px`** (new, scoped exception, D-08). Not one of the seven named tokens above, but still a 4px-grid multiple (4 × 7). Chosen as the tight midpoint of D-08's "roughly 28-32px" range — snug around the existing 20×14 morph icon (28 − 20 = 8px horizontal breathing room / 2 = 4px each side; 28 − 14 = 14px vertical / 2 = 7px each side) without reusing `--space-xl` (32px, already means something else — layout gaps) as a sizing dimension. If live verification shows 28px reads too cramped against the header's other 44px-tall items, step up to 32px — this is a starting point, not an unverifiable mandate (same "verify live, adjust" precedent as Phase 6's Typography section).
- **Instagram nav-link mobile fit** — re-measurement is required (D-03), not a fixed formula. Starting hypothesis, extending the existing pixel-budget precedent (`HomeCarousel.astro:1232-1249`, 393px viewport): the last measured 4-item row used 346px of 361px available (logo 56 + nav "About"+"Contact" 157 + toggle 30 + switcher 79, gaps trimmed to `--space-sm` = 8px × 3). Adding a 5th item — an icon-only Instagram link styled as `.home-nav-link` (padding `var(--space-xs) var(--space-sm)` = 4px/8px around a ~20px-wide glyph ⇒ roughly 36px) plus one more 8px gap — projects to **~390px, ~29px over budget** at this exact viewport. This is a real, expected overflow, not a hypothetical: the plan must re-measure live (same iterative methodology as the original 04.3 mobile-fit work) and close the gap via one or more of: trimming `.home-header`'s gap further, reducing the Instagram glyph's own footprint, or tightening `.home-nav`'s internal gap between About/Contact/Instagram specifically (leave the toggle/switcher untouched, since those are unrelated to this addition). `flex-wrap: wrap` already exists on `.home-header` as a safety net (`HomeCarousel.astro:1246-1249`) — acceptable as a last-resort fallback, but a single-row fit is the intended outcome per D-03's phrasing ("keep the icon in the same single-row flex layout on mobile").

## Layout & Interaction Notes

*(Project convention — see Phase 6's `06-UI-SPEC.md` "Layout & Interaction Notes"; this section is the canonical source for this phase's three fixes.)*

### Instagram icon link (D-01–D-05, HOME-04)

- **Markup position (D-02):** new `<a>` sibling inside `.home-nav`, positioned **after** the existing Contact link and **before** `.home-toggle`: `Logo — About — Contact — Instagram — [toggle] — FR|EN`. It is grouped with the "leave the page" links (About/Contact), not with the toggle/switcher (display controls), matching `.home-nav`'s existing semantic scope.
- **Visual treatment (D-01, D-04):** reuse the `.home-nav-link` class as-is (`display:inline-flex; align-items:center; font-size:14px; padding: var(--space-xs) var(--space-sm); min-height: var(--tap-target-min); color:inherit; text-decoration:none;`) — **no border/box** around the icon (unlike the toggle). The toggle needed a box because a bare icon didn't read as clickable sitting among *plain text* links; Instagram's glyph is expected to read as recognizable/clickable on its own among the *other* plain-text nav links. **Verify live** — if it doesn't read as clickable, add a border box as a follow-up, but do not pre-emptively add one.
- **Glyph (D-01):** standard, widely-recognized Instagram rounded-square-camera-outline glyph, hand-authored as a dependency-free inline `<svg>` using `fill="currentColor"` or `stroke="currentColor"` (matches the "no icon library" constraint already established for the toggle icon). Exact path data is Claude's Discretion — any standard, recognizable rendering is acceptable; size to roughly the same visual footprint as the toggle's 20×14 morph icon (e.g. 20×20, since Instagram's glyph is conventionally square — a stretched/squashed Instagram mark would read as broken, unlike the toggle's intentionally non-square bar/grid glyph).
- **Link behavior (D-05):** exact reuse of the footer's existing attributes/copy source, no re-derivation — `href="https://www.instagram.com/ajs_romanelepont/"`, `target="_blank"`, `rel="noopener noreferrer"`, plus the `instagramNewTabHint` sr-only suffix (see Copywriting Contract above).
- **Mobile fit (D-03):** see the Spacing Scale exception above — this is expected to require live re-measurement and gap/padding trimming; do not treat a first-pass overflow as a bug, treat it as the known next step.

### Toggle square border (D-06–D-09, HOME-05)

- **Root cause (D-06):** `.home-toggle`'s bounding box is not square today because it has no explicit `width`/`height` — its box is whatever the 20×14 icon plus `padding: var(--space-xs)` (4px all sides) plus the 1.5px border happens to produce (≈31px wide × ≈25px tall). This is a box-dimension bug, not a corner-radius bug (corners already read as square — no `border-radius` is set anywhere, matching `--radius-none: 0px`).
- **Structural fix — two nested boxes, not one (new for this phase):**
  1. **Outer `.home-toggle` (the `<button>` itself):** becomes the *invisible, tappable* box. Remove its `border` and `padding` shorthand. Set explicit `min-width: var(--tap-target-min); min-height: var(--tap-target-min);` (currently only `min-height` is set, inherited from the shared rule with `.home-nav-link` — add `min-width` too), keep `display:inline-flex; align-items:center; justify-content:center;` so its content centers.
  2. **New inner wrapping element, e.g. `<span class="home-toggle__box" aria-hidden="true">`,** placed directly around the existing `.home-toggle__morph` icon (inside the button). This is the *visible* square: explicit `width: 28px; height: 28px;` (see Spacing Scale exception), `display:flex; align-items:center; justify-content:center;`, and the border moves here: `border: 1.5px solid currentColor;` (identical value, just relocated from `.home-toggle`).
  3. **Move, don't duplicate, the hover-invert background:** `.home-toggle:hover, .home-toggle:focus-visible { background-color: currentColor; }` (currently at `HomeCarousel.astro:745-748`) must move to target `.home-toggle:hover .home-toggle__box, .home-toggle:focus-visible .home-toggle__box` — painting the full 44px invisible hit-box on hover would visually contradict the "tight compact square" goal this fix exists to achieve.
  4. The existing morph-cell color-invert rules (`.home-toggle:hover .home-toggle__morph-cell`, `HomeCarousel.astro:750-763`) are unaffected — the descendant combinator still resolves correctly through the new intermediate `.home-toggle__box` wrapper.
  5. The attention-pulse animation (`.home-toggle:not(.home-toggle--used)`, `HomeCarousel.astro:793-796`) currently scales `.home-toggle` itself. Recommend moving `transform: scale(...)` to target `.home-toggle__box` instead, so the visible square pulses (scaling an invisible 44px box has no visible effect either way, but scaling the visible square is the intended affordance) — low-risk, minor consistency cleanup, not a hard requirement.
- **Icon proportions untouched (D-07):** `.home-toggle__morph`'s own `width:20px; height:14px;` and the fixed `row-gap:2px`/animated `column-gap` stay exactly as they are — do not resize or reproportion the icon itself to "help" it fit the new square; the square box exists to contain it, not the reverse.
- **Verify in both modes (D-09):** carousel mode and grid mode each have their own `.home-toggle`/`.home-toggle__morph-cell` color overrides (`HomeCarousel.astro:740-763`) — confirm the square border and its hover-invert both render correctly in **both** `data-display-mode="carousel"` and `data-display-mode="grid"`, not just one, per the phase's success criterion #2.

### Mobile hero full-bleed regression (D-10–D-12, HOME-06)

This fix has no new visual design of its own — the acceptance bar is a **regression contract**: restore the exact behavior Phase 6 shipped and live-verified (full-bleed `100svh` photo, no white gap above the header, no footer visible/bleeding through on first load), without removing the mobile carousel/grid morph animation shipped afterward by quick task `260713-jfz`.

- **Investigation order (D-10):** start with the view-transition wiring, not the `100svh`/layout chain from scratch. The prime suspects, in the exact file:
  - `view-transition-name: ajs-hero-morph` on `.home-hero__photo` (`HomeCarousel.astro:888`) and `view-transition-name: ajs-accent-panel` on `.home-hero__accent` (`HomeCarousel.astro:1033`) — both assigned unconditionally (not scoped to desktop), on the same elements whose mobile sizing (`min-height:100svh`, `max-height:100vh`) is the regression's subject.
  - The global `<style is:global>` block (`HomeCarousel.astro:1374+`) — `::view-transition-group(ajs-hero-morph)` z-index stacking, the reduced-motion-scoped `::view-transition-old(root)`/`::view-transition-new(root)` 420ms duration override, and `quick-260713-kit`'s accent-panel fade keyframes. A plausible, concrete hypothesis worth checking first: the View Transition API snapshots the outgoing/incoming DOM state into its own `::view-transition-*` pseudo-element tree, sized from the document/viewport at snapshot time; if that snapshot happens before a mobile browser's dynamic address-bar chrome has finished collapsing on first load (the same `100vh`-vs-`100svh` class of timing bug that already fooled devtools-only testing once, per Phase 6's history), the transition-group pseudo could briefly paint at a stale, taller-than-`100svh` size — which would read exactly as "white gap above header + footer visible" while the mismatch resolves. This is a hypothesis to test, not a confirmed diagnosis.
  - Only after ruling out the above, fall back to re-auditing the `100svh`/`max-height:100vh`/absolutely-positioned-`.home-hero__accent` chain documented in `06-01-SUMMARY.md`'s "Issues Encountered" section from scratch.
- **Resolution constraint (D-12):** if the view-transition hypothesis is confirmed, fix the specific CSS/layout interaction and **keep the morph animation active on mobile too** — do not gate `document.startViewTransition()` behind `pointer:fine`/desktop-only as a shortcut, since that would silently remove a recently-shipped, user-approved mobile feature. Only disable view-transitions on mobile as a last resort if the conflict proves structurally unfixable without disproportionate risk, and if so, flag it explicitly in the plan/summary as a deviation, not a default outcome.
- **Verification method (D-11):** Playwright mobile-viewport emulation (e.g. an iPhone device profile) is the closing bar for this phase — no live on-device checkpoint is required. Explicitly note in the plan that this is the user's chosen bar, not a guarantee the exact real-device symptom (previously only caught via a real iPhone 17 Pro screenshot) is fully gone; flag it as a candidate follow-up quick task if it recurs live post-ship.
- **No new tokens/colors/typography involved** — this is a pure CSS layout/timing fix scoped to the existing `.home-hero__photo`/`.home-hero__accent` mobile rules and the global view-transition block; nothing in the Spacing/Typography/Color tables above changes as a result of this fix.
