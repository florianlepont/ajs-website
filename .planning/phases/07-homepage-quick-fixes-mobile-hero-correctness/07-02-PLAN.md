<objective>
Root-cause and fix the mobile full-bleed hero regression (HOME-06): on a real phone (reported on iPhone 17 Pro), on first load the homepage shows a white gap above the header plus the site footer bleeding through beneath the hero — a regression against Phase 6's live-verified `100svh` full-bleed hero.

Per D-10, investigate the View Transitions wiring FIRST (added by quick task `260713-jfz` AFTER Phase 6's `100svh` fix was last verified working, never re-tested on a real device), before re-auditing the `100svh`/`max-height:100vh`/absolute-panel chain from scratch. Per D-12, fix the specific CSS/layout interaction and KEEP the morph animation active on mobile — do not scope `document.startViewTransition()` to desktop as a shortcut. Per D-11, close the phase with Playwright mobile-viewport emulation only (no live on-device checkpoint).

Purpose: The mobile hero is the first thing a phone visitor sees; the regression undermines the full-bleed impression Phase 6 shipped.
Output: A new mobile-emulation regression test in `tests/e2e/homepage.spec.ts` plus the targeted CSS fix in `HomeCarousel.astro`.

## Phase Goal (user story)

**As a** visitor on a phone, **I want to** see the homepage hero fill my screen edge-to-edge on first load, **so that** Romane's photography lands immediately with no white gap above the header and no footer bleeding through beneath it.

> Note (MVP-mode judgement): this is a regression fix, so the natural vertical slice is test-first — Task 1 encodes the full-bleed invariants as a failing/guarding mobile-emulation test, Task 2 root-causes and applies the CSS fix to satisfy it. The user story is synthesised from the ROADMAP success criterion #3 for HOME-06; no capability was invented.

> Note (D-11 verification bar): emulation is the user's explicitly chosen closing bar for this phase, NOT a guarantee the exact real-device symptom is gone. This bug class (`100vh` vs `100svh` Safari-chrome timing) has already escaped devtools/emulation testing once before (Phase 6's fix was only caught via a real iPhone 17 Pro screenshot), and the Playwright suite runs the iPhone profile on the Chromium engine (viewport + isMobile + hasTouch emulation), which is further from real mobile Safari than a WebKit profile would be. If the symptom recurs live post-ship, flag it as a candidate follow-up quick task — do not treat green emulation as proof the real-device bug is impossible.
</objective>
