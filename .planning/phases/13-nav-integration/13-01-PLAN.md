<objective>
Wire a new "Éditions" entry into the shared `<SiteHeader>` component so visitors discover the Phase 12 Éditions overview page from the main nav on EVERY page (homepage, gallery pages, About, Contact), in both French and English, with the label editable by Romane via `siteSettings.navLabels` — exactly mirroring the already-shipped `about`/`contact` label pattern. This is the sole deliverable of EDN-01.

Purpose: Give Éditions top-level nav discoverability (D-01: first link, highest visual priority) while keeping the homepage's photography carousel/grid pure photography (D-13 / SC #3 — nav only, never carousel/grid data).

Output: A 4th nav link ("Éditions") rendered first in `.site-nav` at both `<SiteHeader>` call sites, a bilingual `navLabels.editions` Sanity field, an `editionsLabel` resolver key, and re-measured mobile-fit CSS that keeps the 4-link header on one row down to the narrowest supported phone (D-02/D-03).
</objective>

## Phase Goal

**As a** site visitor, **I want to** find an "Éditions" link in the main navigation on every page (in my current language), **so that** I can reach the Éditions overview and browse Romane's paper éditions from anywhere on the site.

> Story derivation note: the ROADMAP Phase 13 `**Goal:**` line is outcome-shaped but not in `As a / I want to / so that` form. This story is derived verbatim from requirement EDN-01 ("Visitor can discover an 'Éditions' section via a new top-level main-nav entry") plus the ROADMAP goal's "on every page … stays pure photography" boundary — nothing invented.

## Assessment: threat surface is negligible for this phase

Per the security contribution guidance for Phase 13: this phase adds one presentational navigational `<a>` link and one bilingual plain-text Sanity schema field. It introduces **no new request-time surface** (the site is static, zero-compute on OVH/GitHub Pages — all Sanity content is fetched at BUILD time, published perspective only), **no new user input**, **no new authentication or authorization**, **no new sensitive-data storage**, and **no package installs** (npm/pip/cargo). A full STRIDE sweep would be fabricating threats that do not apply; the single genuinely-relevant consideration is content-injection via the CMS-authored label, which is already mitigated by the identical existing about/contact pattern.
