# Phase 04.1: Design System & Homepage Refresh - Context

**Gathered:** 2026-07-08
**Status:** Ready for planning

<domain>
## Phase Boundary

The site adopts a new monochrome + pop-pink visual identity (imported via Claude Design), superseding Phase 2's Dawn Pink/Woodsmoke/Wild Strawberry palette — applied across every already-shipped page (portfolio galleries, about, contact, legal pages), not just new work. The homepage is rebuilt from its current "Bienvenue... en cours de construction" placeholder into a real gallery entry point: a hero carousel (one migrated gallery at a time, auto-advancing, rotating decorative accent color) with a toggle to a 2-column grid view. New reusable form/feedback components (Button, Input, Textarea, EmptyState) are introduced and given real first usages. This phase does not cover exhibitions, shop/checkout (v1.x, out of scope), or the Phase 5 domain cutover.

</domain>

<decisions>
## Implementation Decisions

### Rebrand Scope
- **D-01:** Full site rebrand now — repaint all already-shipped pages (portfolio galleries, about, contact, legal/mentions-légales/confidentialité) to the new identity in this phase, not just new work. Avoids a visually split site before launch.
- **D-02:** Remove the dormant Delight `@font-face` declaration in `BaseLayout.astro` — it never actually rendered (fallback-only, no font file was ever sourced/licensed) and Archivo Black replaces its role as the Heading/Display typeface. Also remove its tracking entry in `deferred-items.md`.
- **D-03:** Adopt the imported design system's ink-only error convention for form validation — remove `--color-destructive` (#dc2626) red entirely; error states use bold ink text + a thicker ink border instead. Red would otherwise be the only non-palette color left on the site.
- **D-04:** Remove `--color-secondary` (#E4D9D0) chrome-band background entirely. Header and footer become white-on-white (`--color-dominant`), delineated only by a 1px hairline border (`--color-border`), matching the imported design system exactly.
- **D-05:** The decorative secondary palette (`--palette-purple` #AF3DFF, `--palette-teal` #55FFE1, `--palette-lime` #A6FD29, `--palette-plum` #37013A) is homepage-hero-carousel-only, used as the rotating accent-panel background per gallery. It is NOT used anywhere else on the site — pink (`--color-accent` #FF3B94) remains the only functional/interactive color (links, buttons, focus rings, current-locale emphasis) everywhere outside the carousel.
- **D-06:** Links render in full pink text (`--color-accent`), not the current ink-text-plus-pink-underline pattern. This applies to nav links, body copy links, and current-locale emphasis site-wide — per the imported design system's explicit rule that link text itself reads in pink now.
- **D-07:** Adopt the full-bleed-hero-photo-with-dark-gradient-overlay technique (for white text legibility) on the existing gallery-detail page header too, not just the new homepage — keeps the visual language consistent between the homepage carousel and gallery-detail pages. This is a new technique not previously in the codebase.

### Homepage Interaction Model
- **D-08:** Build BOTH homepage display modes from the prototype: a full-screen hero carousel AND a 2-column grid view, with a toggle between them (matches the prototype exactly — "Carrousel"/"Carousel" and "Grille"/"Grid" labels).
- **D-09:** The hero carousel auto-advances between galleries on a timer, pausing on hover/interaction. (Note: this is a deliberate departure from the imported design system's own "motion-neutral by default" guidance — user explicitly chose auto-advance over manual-only.)
- **D-10 (Claude's Discretion):** Mobile responsive behavior for the carousel's desktop-oriented layout (full-viewport hero, ~52%-width bottom-right accent panel) is left to the planner/executor — no strong preference stated beyond "stack vertically, full-width accent panel" being the suggested default.

### Logo & Imagery
- **D-11:** Adopt the real AJS wordmark PNG logo in the header, replacing the current text-only site-title. Scope for this phase is **header only** — not the footer, not the favicon.
- **D-12:** The homepage hero carousel and grid show ONLY galleries that already have real migrated photos (currently: Silos, Brume — 2 of 6 planned galleries per Phase 2's deferred scope). Unmigrated galleries (Rebut, Adults, The Victorian Tea Room, Paysages) are excluded from the new homepage until they get real photos — avoids placeholder/broken-looking hero images on a highly visual homepage. (Note: this does not affect the separate `/galleries/` listing page, which is out of this phase's scope and continues to show whatever `getGalleries()` returns.)
- **D-13:** The header logo has a hover effect carried over from the current live site (atelierjacquelinesuzanne.fr): hovering inverts the logo's colors by swapping between two logo image variants. The imported asset pack conveniently already includes both halves of this pair — `AJS_Brutalist_Black_Transparent.png` and `AJS_Brutalist_White_Transparent.png` — implement the hover as a swap between these two.

### New Form Components
- **D-14:** Refactor `ContactForm.astro` (Phase 3, already shipped and working) to use the new Input/Textarea/Button component patterns. Low risk — e2e coverage already exists to catch regressions, and D-03's ink-only error style already requires touching this form's error-state markup.
- **D-15:** The Button component's first real usage in this phase is the contact form's submit button (the homepage's only CTA is a plain underlined text link per the prototype, not a filled button — so Button isn't exercised there).
- **D-16:** Refactor the galleries listing page's existing inline empty-state ("Galeries à venir" / "Galleries coming soon") to use the new EmptyState component, giving it a real usage immediately.

### Claude's Discretion
- Mobile responsive behavior for the hero carousel layout (see D-10).
- Exact pixel/spacing translation from the prototype's inline styles into real component CSS — the prototype is a click-through reference, not literal implementation code.
- Whether the intro/welcome copy panel in the carousel reuses the existing `siteSettings.welcomeHeading`/`welcomeBody` Sanity fields or is hardcoded — not discussed; use existing Sanity-driven content where a natural fit exists, per the established pattern.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `GalleryCard.astro` already implements the `transform: scale(1.02)` hover treatment the imported design system specifies — no behavior change needed, only token repaint.
- `Lightbox.astro` is a dependency-free vanilla-JS island built on native `<dialog>` (`.showModal()`) — solves focus containment and Escape-to-close natively. The imported `Lightbox.jsx` reference uses Unicode `‹ › ✕` glyphs as its icon placeholder; keep that glyph choice (matches the "no icon library" convention) but do not replace the underlying `<dialog>` architecture.
- `getGalleries()` in `src/lib/sanity.ts` already returns galleries in Romane's manual order (D-10 from 02-CONTEXT.md) — the homepage carousel/grid should call this directly, filtered to only entries with real images (D-12), not introduce a new query function.
- `border-radius: 0` is already the convention in the one place it appears (`ContactForm.astro`) — square corners are not a new constraint, already aligned with the imported design system.

### Established Patterns
- Page pairs live at `src/pages/{slug}.astro` (FR) / `src/pages/en/{slug}.astro` (EN), same structure, only import depth and locale key differ — the new homepage follows this exact pattern.
- No UI framework dependency anywhere (`output: 'static'`, OVH zero-compute constraint) — the imported design system's components are React (`.jsx`); every one must be translated to plain Astro + vanilla-JS islands (matching `LanguageSwitcher.astro`'s and `Lightbox.astro`'s existing island pattern), not introduced as literal React code.
- Locale-object content (`{fr, en}` fields via Sanity) vs. hardcoded locale-conditional strings — legal pages and nav labels use hardcoded conditionals; gallery/site-settings content comes from Sanity. New homepage copy (intro text, CTA label) should follow whichever pattern matches its content type (see Claude's Discretion note above).

### Integration Points
- `BaseLayout.astro`'s footer already has two `<nav>`-style legal links (Phase 4) using the existing `.nav-link` CSS class — the D-11 logo swap and D-04 chrome restructure touch this same file's header/footer markup.
- Homepage carousel/grid state (display mode, current index, locale) is currently simulated in the prototype's `DCLogic` component class — this needs translation to a vanilla-JS island (state via `data-*` attributes or a small class, matching `LanguageSwitcher.astro`'s script-tag pattern), not a framework component.

</code_context>
