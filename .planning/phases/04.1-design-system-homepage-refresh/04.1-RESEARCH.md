## Summary

This phase is a repaint + one net-new page, not a new architecture. The live codebase (Astro 7.0.6,
`output: 'static'`, no UI framework, no SSR adapter) already has every structural piece the imported
design system needs: a single `:root` token block in `src/layouts/BaseLayout.astro`, a
component-per-file convention with FR/EN page-pair mirroring, and two working vanilla-JS islands
(`LanguageSwitcher.astro`, `Lightbox.astro`) that establish the exact pattern the new homepage
carousel/grid island and the new Button/Input/Textarea/EmptyState components must follow. Nothing
here requires installing a UI framework, a carousel library, or a CMS schema change — the homepage's
data (`getGalleries()`) already exists and already returns Romane's manually-ordered galleries.

The bulk of the work is: (1) swap the `:root` token block from Dawn Pink/Woodsmoke/Wild Strawberry to
the imported monochrome + pop-pink tokens, deleting `--color-secondary`'s tint and `--color-destructive`
entirely per CONTEXT.md D-03/D-04; (2) self-host Archivo Black (the design system's own token file
points at Google Fonts CDN, which is a real GDPR/CNIL data-transfer concern for this EU/France-focused,
already-legally-scrubbed site — see Pitfall 1); (3) build four small, dependency-free Astro components
(Button, Input, Textarea, EmptyState) translated from the design system's React reference `.jsx` files
into plain Astro markup/CSS, no hydration needed since none of them require client-side state of their
own; (4) build one genuinely new island — the homepage's carousel/grid state machine — as a vanilla-JS
`<script>` block exactly matching `LanguageSwitcher.astro`'s existing pattern (no `client:*` directive,
because there's no framework component to hydrate); (5) repaint five existing files
(`ContactForm.astro`, `GalleryCard.astro`, `GalleryGrid.astro`, `Lightbox.astro`,
`LanguageSwitcher.astro`) that already implement the correct *structure* and only need token/color
touch-ups, not rewrites.

**Primary recommendation:** Treat `BaseLayout.astro`'s `:root` block as the single point of change for
colors/fonts (already proven safe in Phase 2's rebrand — no e2e test asserts on hex values, only on
accessible roles/text), build the four new form/feedback components as plain `.astro` files with
scoped `<style>` blocks reading the same CSS custom properties, and implement the homepage as a single
new island (`HomeCarousel.astro` or similar) whose state (display mode, carousel index, locale) lives
in a small class-free `<script>` closure keyed off `data-*` attributes — mirroring the prototype's
`DCLogic`/`renderVals()` state shape one-to-one, since that JS is effectively already a working spec
for the state machine's transitions.

A visitor's primary path (input → output): Astro build reads Sanity → emits static homepage HTML with

the initial page load(s) — everything is prerendered static HTML plus client-side DOM manipulation.

### Pattern 1: Self-hosting Archivo Black via `@fontsource`

**What:** Replace the imported `tokens/fonts.css`'s `@import url('https://fonts.googleapis.com/...')`
with an npm-installed, locally-bundled font package, imported once in `BaseLayout.astro`'s frontmatter
(not a `<link>` tag) so Vite/Astro bundles the woff2 file into the build output automatically.

**When to use:** Any time a design import specifies a Google Fonts CDN `@import`/`<link>` for an
EU/France-facing site — CNIL guidance and a documented 2022 German court ruling (LG München I) treat
loading fonts from Google's CDN as an unlawful transfer of the visitor's IP address to the US without
consent, distinct from *using* a Google-published font file. `[ASSUMED]` — CNIL/case-law claims here
are based on general knowledge, not fetched from an official CNIL page in this session; flag for the
planner to spot-check against current CNIL guidance if this becomes contentious, but self-hosting is
the safe default regardless (it also removes a render-blocking third-party request, a performance win
regardless of the legal question).

**Example:**
```astro
---
// src/layouts/BaseLayout.astro frontmatter
import '@fontsource/archivo-black'; // ships woff2 + @font-face, no Google CDN request
---
<style is:global>
  :root {
    --font-display: 'Archivo Black', var(--font-sans);
  }
</style>
```

### Pattern 2: Translating a design-system `.jsx` reference component to Astro

**What:** Each `components/forms/*.jsx` file in the design import is a *reference*, not drop-in code —
it's plain React with inline `style` objects. The imported `Button.jsx`/`Input.jsx`/`Textarea.jsx` have
no internal state (no `useState`, no effects) — they are pure presentational functions, which makes the
Astro translation mechanical: props become `Astro.props`, inline `style` objects become scoped
`<style>` blocks reading the same CSS custom properties, and `{...rest}`/`{children}` become Astro's
native prop spreading and `<slot />`.

**When to use:** For all four new components (Button, Input, Textarea, EmptyState) — none require
client-side interactivity of their own, so none need a `client:*` hydration directive or even a
`<script>` tag. `ContactForm.astro`'s existing `<script>` (validation/submit logic) continues to own
all runtime behavior; the refactor only changes what markup that script queries (e.g.
`form.querySelector('[data-role="submit-button"]')` still works if the Button component preserves that
`data-role` attribute via a prop or hardcodes it internally for this specific usage).

**Example (Button.astro, translated from Button.jsx):**
```astro
---
interface Props {
  variant?: 'primary' | 'secondary';
  size?: 'sm' | 'md';
  disabled?: boolean;
  type?: 'button' | 'submit' | 'reset';
}
const { variant = 'primary', size = 'md', disabled = false, type = 'button' } = Astro.props;
---
<button type={type} disabled={disabled} class={`btn btn--${variant} btn--${size}`}>
  <slot />
</button>

<style>
  .btn {
    font-family: var(--font-sans);
    font-size: var(--text-label-size);
    font-weight: var(--weight-semibold);
    padding: var(--space-sm) var(--space-lg);
    min-height: var(--tap-target-min);
    border-radius: var(--radius-sm);
    cursor: pointer;
    display: inline-flex;
    align-items: center;
    justify-content: center;
    gap: var(--space-xs);
    transition: opacity 120ms ease;
  }
  .btn:disabled { cursor: not-allowed; opacity: 0.4; }
  .btn--primary {
    background: var(--color-accent);
    color: var(--color-on-accent);
    border: 1px solid var(--color-accent);
  }
  .btn--secondary {
    background: transparent;
    color: var(--color-accent);
    border: 1px solid var(--color-accent);
  }
  .btn:not(:disabled):hover { opacity: 0.8; }
</style>
```
*(Source: translated from `.planning/design-import/design-system/components/forms/Button.jsx`, tokens
per `.planning/design-import/design-system/tokens/*.css`)*

### Pattern 3: CSS-only logo hover-invert (D-13)

**What:** Two stacked `<img>` elements (or one `<img>` + a `background-image` layer) inside a
positioned wrapper; the "white" variant is `opacity: 0` by default and `opacity: 1` on
`:hover`/`:focus-within`, cross-fading via `transition: opacity`.

**Example:**
```astro
<a href={homeHref} class="logo-mark" aria-label={siteTitle}>
  <img src="/logos/AJS_Brutalist_Black_Transparent.png" alt="" class="logo-mark__img logo-mark__img--default" />
  <img src="/logos/AJS_Brutalist_White_Transparent.png" alt="" class="logo-mark__img logo-mark__img--hover" />
</a>

<style>
  .logo-mark { position: relative; display: inline-block; }
  .logo-mark__img { display: block; width: 48px; height: auto; }
  .logo-mark__img--hover {
    position: absolute; inset: 0; opacity: 0; transition: opacity 150ms ease;
    /* needs a dark backdrop behind the white variant to be visible on hover —
       see note below */
  }
  .logo-mark:hover .logo-mark__img--hover,
  .logo-mark:focus-within .logo-mark__img--hover { opacity: 1; }
</style>
```
**Caveat:** the white PNG variant is only visible against a dark background. Since D-04 makes the
header white-on-white, the hover-invert needs either (a) a small dark backdrop chip behind the logo on
hover (matching the live site's current behavior, per D-13's "carried over from the live site" note —
the researcher could not inspect the live site's exact implementation in this session, so this is
`[ASSUMED]`; confirm the live site's actual hover treatment before implementing), or (b) reinterpret
"invert" as swapping to the black-background lockup with a small inline dark chip. **Flagged as an
Open Question below** — the planner/executor should verify against the live
atelierjacquelinesuzanne.fr site's actual hover behavior (mentioned as already confirmed by the user

not specified in CONTEXT.md).

### Pattern 4: Homepage carousel/grid state machine (vanilla JS, no framework)

**What:** A single `<script>` block (like `LanguageSwitcher.astro`'s), reading gallery data from
hidden `data-*`-bearing DOM nodes (like `Lightbox.astro`'s `<ul hidden data-role="lightbox-data">`
pattern), holding a small in-memory state object, and re-rendering by toggling `hidden`/class names —
no virtual DOM, no reactive framework.

**Example (state shape mirrors the prototype's `DCLogic` almost exactly):**
```typescript
// Inside HomeCarousel.astro's <script> tag
interface GalleryEntry { slug: string; title: string; coverSrc: string; coverAlt: string }

const root = document.querySelector<HTMLElement>('[data-role="home-carousel"]');
const dataEl = document.querySelector<HTMLUListElement>('ul[data-role="carousel-data"]');
if (root && dataEl) {
  const galleries: GalleryEntry[] = Array.from(dataEl.querySelectorAll('li')).map((li) => ({
    slug: li.dataset.slug ?? '',
    title: li.dataset.title ?? '',
    coverSrc: li.dataset.src ?? '',
    coverAlt: li.dataset.alt ?? '',
  }));

  const ACCENT_VARS = ['--color-accent', '--palette-purple', '--palette-teal', '--palette-lime', '--palette-plum'];
  let displayMode: 'carousel' | 'grid' = 'carousel';
  let carouselIndex = 0;
  let timer: ReturnType<typeof setInterval> | null = null;

  function render() { /* update hero <img src>, accent panel background, index label */ }
  function startAutoAdvance() {
    timer = setInterval(() => { carouselIndex = (carouselIndex + 1) % galleries.length; render(); }, 6000);
  }
  function stopAutoAdvance() { if (timer) clearInterval(timer); timer = null; }

  root.addEventListener('mouseenter', stopAutoAdvance);
  root.addEventListener('mouseleave', startAutoAdvance);
  root.addEventListener('focusin', stopAutoAdvance);
  root.addEventListener('focusout', startAutoAdvance);

  startAutoAdvance();
  render();
}
```
*(Source: state shape adapted from `.planning/design-import/homepage-prototype/Atelier Homepage.dc.html`'s
`Component` class — `state`, `carouselPrev`/`carouselNext`, `ACCENTS` array — translated from React
`setState` semantics to a plain closure + manual re-render.)*

### Anti-Patterns to Avoid

- **Do not import the design system's `.jsx` files directly or install React/Preact for this phase.**
  CONTEXT.md and CLAUDE.md are both explicit: no UI framework exists or should be introduced; every
  `.jsx` file is a reference to translate, not code to run.
- **Do not replace `Lightbox.astro`'s native `<dialog>` architecture** with the design system's
  `Lightbox.jsx` reference (CONTEXT.md explicitly forbids this — repaint only).
- **Do not put the Sanity fetch inside the client-side carousel script.** `getGalleries()` must run
  once at build time in `index.astro`'s frontmatter (per the existing `SANITY_API_READ_TOKEN`
  never-reaches-browser constraint) — pass the already-filtered (D-12), already-fetched data to the
  island via embedded static markup/data attributes, exactly like `Lightbox.astro` does today.
- **Do not use `setState`/React patterns in the vanilla island.** Keep the re-render function simple
  and imperative (direct DOM mutation), matching `Lightbox.astro`'s `render()` function precedent.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|--------------|-----|
| Modal/lightbox focus containment | A custom focus-trap + Escape-key handler | Native `<dialog>`.`showModal()` (already done in `Lightbox.astro`) | Already solved correctly in the codebase; this phase only repaints its colors, never touches this logic (explicit CONTEXT.md instruction) |
| Font loading | Manually downloading/hosting individual woff2 files and hand-writing `@font-face` | `@fontsource/archivo-black` (npm package bundling both) | Handles subsetting, correct `font-display`, and multiple format fallbacks without hand-maintaining font files in `public/` (contrast with the still-unresolved Delight font situation this phase removes — D-02) |
| Carousel auto-advance/pause-on-hover | A hand-rolled requestAnimationFrame loop | `setInterval`/`clearInterval` on hover/focus, exactly as sketched in Pattern 4 | The interaction is simple enough (single timer, no easing/animation-frame precision needed) that `setInterval` is the right level of complexity — a rAF loop would be solving a problem this phase doesn't have (no smooth continuous-scroll requirement was specified) |
| Image responsive sizing/format conversion | Manual `<picture>`/`srcset` hand-authoring per breakpoint | Astro's built-in `astro:assets` `<Image />` component, IF the phase decides to adopt it — **note:** the current codebase does NOT use `astro:assets` yet (galleries render plain `<img>` tags built from Sanity CDN URLs via `src/lib/image.ts`'s `thumbnailUrl`/`fullSizeUrl` helpers) — this is an existing, working pattern; do not introduce `astro:assets` mid-phase for the homepage hero image alone, as it would create an inconsistent optimization strategy across gallery pages vs. the homepage. and one carousel state machine — resist the urge to introduce new abstractions (a generic "Carousel"
component library, a design-token build pipeline, etc.) for a single-instance, single-page feature.

## Common Pitfalls

### Pitfall 1: Google Fonts CDN `@import` is a GDPR/CNIL exposure for this specific project
**What goes wrong:** Copying `tokens/fonts.css` verbatim (`@import url('https://fonts.googleapis.com/...')`)
loads a third-party (Google) resource on every page view, transmitting the visitor's IP address to
Google's servers without the user's knowledge — for most sites a minor concern, but this project's
Phase 4 privacy policy (`src/pages/confidentialite.astro`, already shipped and human-signed-off) makes
an explicit, specific claim: *"le contenu du site... provient d'un système de gestion de contenu
(Sanity)... Aucune donnée vous concernant n'est jamais transmise à [un tiers]"* and enumerates exactly
three data flows (contact form → OVH mailbox, Sanity content fetch at build time, OVH hosting logs).
Adding a Google Fonts CDN request introduces an undisclosed fourth data flow to a legal document a
human already signed off on.
**Why it happens:** The imported design system's token file was generated without awareness of this
project's already-shipped privacy policy specifics (it predates Phase 4 entirely, per the design-import
README's own staleness caveat).
**How to avoid:** Self-host via `@fontsource/archivo-black` (Pattern 1) instead of the CDN `@import`.
**Warning signs:** Any `<link rel="stylesheet" href="https://fonts.googleapis.com/...">` or `@import
url('https://fonts.googleapis.com/...')` surviving into the shipped `BaseLayout.astro`.

### Pitfall 2: `--color-destructive` removal breaks `ContactForm.astro`'s inline JS, not just its CSS
**What goes wrong:** D-03 removes `--color-destructive` (#dc2626) entirely, but
`ContactForm.astro`'s `<script>` block currently does `statusEl.style.color =
'var(--color-destructive)'` in its `renderSubmissionError()` function (an inline style assignment, not
a CSS class) — a pure CSS-file token swap will silently leave this JS-set inline style pointing at an
undefined custom property (which the browser will simply not apply, likely falling back to inherited
black/ink text — probably fine visually by accident, but not intentional and untested).
**Why it happens:** The color is set imperatively in JS, not declared in the component's `<style>`
block where a token-swap sweep would naturally catch it.
**How to avoid:** When refactoring `ContactForm.astro` for D-14, grep the file's `<script>` block (not
just its `<style>` block) for `--color-destructive` and replace the submission-error styling with the
D-03 ink-only convention (e.g. toggle a CSS class like `.contact-form__status--error` that sets bold
font-weight + ink color, rather than assigning `style.color` directly).
**Warning signs:** `grep -rn "color-destructive" src/` should return zero results after this phase
ships — currently it appears in both `ContactForm.astro`'s `<style>` (2 occurrences) and its `<script>`
(1 occurrence, `renderSubmissionError`).

### Pitfall 3: `LanguageSwitcher.astro`'s hardcoded hex color is a coincidental non-issue, but still worth cleaning up
**What goes wrong/opportunity:** `LanguageSwitcher.astro` hardcodes `color: #1a1a1a` (not
`var(--color-ink)`) in three places. This was written before Phase 2 shipped Woodsmoke (`#141213`), so
it's actually been slightly wrong (not matching the shipped ink token) since Phase 2 — but by
coincidence, the *new* rebrand's ink token (`#1A1A1A`) matches this hardcoded value exactly. No visual
bug will occur if this file is left completely untouched. However, CONTEXT.md's `<canonical_refs>`
lists this file as a "repaint" target, implying the planner intends real token-reference edits here;
recommend replacing the hardcoded hex with `var(--color-ink)` while touching the file anyway, for
future-proofing (so a subsequent rebrand doesn't require remembering this file separately).
**How to avoid:** Explicit grep-and-replace of literal hex colors with `var(--color-ink)` across all
"repaint only" files, not just a visual smoke-test (a hardcoded value that happens to match will pass
any visual check while still being technical debt).

### Pitfall 4: Archivo Black is a single-weight font; the type-role contract asks for a semibold (600) Heading/Display
**What goes wrong:** `tokens/typography.css` declares `--text-heading-weight: 600` and
`--text-display-weight: 600`, both mapped to `--font-display` (Archivo Black). Archivo Black ships as a
single static weight (400 in its own font-weight metadata, despite its visually "black"/bold
appearance) — browsers will synthesize/fake-bold a `font-weight: 600` request against a font file that
has no 600 weight variant, which most browsers handle by just rendering the available weight (visually
fine, since Archivo Black already looks bold) but is worth a visual sanity check rather than assuming
it's a no-op.
**Why it happens:** Type-role tokens were designed as an abstract 2-weight system (regular/semibold)
applied uniformly across both the sans body font (which DOES have real 400/600 weights available in
system fonts) and the display font (which doesn't).
**How to avoid:** After wiring `--font-display`, do a visual check of an `<h1>`/Display-role element —
if the browser is silently ignoring the `font-weight: 600` declaration (common with single-weight
webfonts) rather than complaining, this is a non-issue and no action is needed; only intervene if a
visual regression (unexpectedly thin heading text) appears.
**Warning signs:** Headings rendering visually lighter than expected after the font swap.

### Pitfall 5: `image-slot.js`/`_ds_bundle.js`/`support.js` in the design-import folders are prototype-only tooling, not shippable code
**What goes wrong:** The `homepage-prototype/` and `design-system/` folders contain
`_ds_bundle.js`, `support.js`, `image-slot.js`, and the `<x-dc>`/`<x-import>`/`sc-if`/`sc-for` custom
elements used only to render the Claude-Design click-through preview in a browser. None of this is
meant to ship — it's tooling for viewing the prototype, not a library to adopt.
**How to avoid:** Only extract the *visual/interaction intent* (markup structure, inline styles, the
`GALLERIES`/`ACCENTS`/`Component` class's state transitions) from `Atelier Homepage.dc.html` — never
`import` or reference `_ds_bundle.js`, `support.js`, or `image-slot.js` from real site code.
**Warning signs:** Any reference to `x-dc`, `x-import`, `sc-if`, `sc-for`, or a script tag pointing at
`_ds/`/`support.js`/`image-slot.js` inside `src/`.
