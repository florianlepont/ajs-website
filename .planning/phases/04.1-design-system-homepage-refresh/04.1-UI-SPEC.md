## Spacing Scale

Declared values (must be multiples of 4) — **unchanged from Phase 1/2**, confirmed identical
between the live codebase and the imported design system's `tokens/spacing.css` (design-import
README: "no migration needed there, only color and font tokens change"):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon/inline gaps (e.g. carousel prev/next icon padding, form field label-to-input gap) |
| sm | 8px | Compact spacing (switcher padding, footer line spacing, form field internal gap) |
| md | 16px | Default spacing (nav gaps, paragraph spacing, input padding) |
| lg | 24px | Section padding, header/footer vertical padding (mobile), Button padding (primary) |
| xl | 32px | Layout gaps, header/footer vertical padding (desktop), hero carousel chrome padding |
| 2xl | 48px | Major section breaks, EmptyState vertical padding |
| 3xl | 64px | Page-level spacing, EmptyState outer padding |

Exceptions:
- **44px minimum tap target** (`--tap-target-min`) — applies to: Button component (all sizes/variants), Input/Textarea fields, lightbox controls (unchanged), and the new homepage carousel's prev/next arrows and carousel/grid toggle buttons. Applied via padding/`min-height`, never by inflating font-size (WCAG 2.5.5).
- **Radii:** `--radius-none: 0px` (chrome, cards, panels — square corners, brutalist wordmark cue, already the convention in `ContactForm.astro`) and `--radius-sm: 2px` (Button/Input/Textarea only — minimal radius, touch-affordance softening only, never applied to cards/chrome/panels).
- **Border widths:** `--border-hairline: 1px` (default input/card borders, header/footer delineation) and `--border-focus: 2px` (Input/Textarea error-state border, per D-03's ink-only convention — see Color section).
- No other exceptions.

## Typography

Exactly 4 roles, exactly 2 weights — unchanged shape from Phase 1/2, new family reference for Heading/Display:

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Label | 14px | 400 (regular) | 1.5 |
| Body | 16px | 400 (regular) | 1.5 |
| Heading | 20px | 600 (semibold) | 1.2 |
| Display | 32px (site-wide fixed value — **not** the fluid `clamp()` mechanism from Phase 2's superseded identity) | 600 (semibold) | 1.2 |

Font family: Label/Body → `var(--font-sans)` (system stack). Heading/Display → `var(--font-display)` = `'Archivo Black', var(--font-sans)`.

**Homepage-specific exception (per the prototype, `Atelier Homepage.dc.html` line 84/110):** the
hero carousel's and grid view's "Atelier Jacqueline Suzanne" wordmark-style intro text renders at
`font-size: clamp(32px, 5vw, 56px); font-weight: 400; line-height: 1.05;` — a deliberately larger,
fluid, lighter-weight treatment than the standard Display role, reserved for this single homepage
intro-panel usage only (matches the prototype exactly, per D-08's "matches the prototype exactly"
instruction). Do not reuse this fluid rule anywhere else; every other Display-role instance
(gallery-detail `<h1>`, 404 heading treated as Heading, etc.) uses the fixed 32px/600/1.2 rule above.

**Pitfall 4 (font-weight synthesis) — visual-check only, not a blocking issue:** Archivo Black ships
as a single static weight; browsers requesting `font-weight: 600` against it typically just render
the available weight (already visually bold) rather than faking a heavier one. Do a visual sanity
check of a rendered `<h1>`/Display element after wiring the font; only intervene if headings look
unexpectedly thin.

## Color

Tokens per `.planning/design-import/design-system/tokens/colors.css` (authoritative — resolves the
readme.md's internally-inconsistent accent hex in favor of the CSS file, per the design-import
README's resolution rule).

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#FFFFFF` | Page background, main content area, **and** header/footer chrome bands (merged, D-04 — see note below). |
| Secondary (30% in the classic split — **merged into Dominant this phase, D-04**) | `#FFFFFF` (alias of Dominant, `--color-secondary: var(--gray-0)`) | Formerly a distinct chrome-band tint (`#E4D9D0` in the superseded Phase 2 identity). Now identical to Dominant — header and footer read as white-on-white, delineated **only** by a 1px hairline border (`--color-border: #E3E1DE`), never by a background-color difference. This is an intentional, locked deviation from a literal 60/30/10 *distinct-color* split — the imported design system's explicit rationale (readme.md "Visual Foundations") is that chrome should recede entirely in favor of imagery and the accent color. |
| Accent (10% in the classic split — **intentionally the most-used functional color this phase, exceeding 10%, per the imported design system's explicit rationale**) | `#FF3B94` (pop pink, `--pink-600`) | Reserved for: nav links (header + footer, body-copy links) rendered as **full pink text** (D-06 — not ink-text-plus-underline), current-locale emphasis in `LanguageSwitcher`, the Button component's primary-variant fill, `:focus-visible` outlines/rings site-wide, the homepage hero carousel's default (index-0) accent panel background. **Never used for:** body/heading text color, card or chrome backgrounds outside the carousel accent panel, decorative fills elsewhere on the site. |
| Destructive | **none — removed entirely (D-03)** | `--color-destructive` (`#dc2626`) is deleted from `BaseLayout.astro`'s `:root` block and from every reference to it (grep target: `ContactForm.astro`'s `<style>` block, 2 occurrences, **and** its `<script>` block's `renderSubmissionError()` inline `style.color` assignment, 1 occurrence — RESEARCH.md Pitfall 2). Error states use **bold ink text** (`var(--color-ink)`, weight 600) + a **thicker ink border** (`var(--border-focus)`, 2px, `var(--color-ink)`) instead of a semantic red. Applies to: Input/Textarea error state, ContactForm's per-field validation errors, and its submission-failure status message. |

**Text/ink color:** `--color-ink: #1A1A1A` — all body copy, headings, nav text (except link color,
which is full pink per D-06), form labels/values.

**Text-on-accent:** `--color-on-accent: #1A1A1A` (dark ink, **not** white) — any text rendered
directly on an accent-pink fill (Button primary-variant label, homepage carousel accent-panel
default-index intro text) uses ink, never white; pink-on-white passes readably, white-on-pink does
not (design-import tokens/colors.css comment).

**Decorative palette — homepage hero-carousel-only, never functional/interactive (D-05):**

| Token | Value | Usage |
|-------|-------|-------|
| `--palette-purple` | `#AF3DFF` | Carousel accent-panel background, gallery index 1 (`ACCENTS[1]`) |
| `--palette-teal` | `#55FFE1` | Carousel accent-panel background, gallery index 2 |
| `--palette-lime` | `#A6FD29` | Carousel accent-panel background, gallery index 3 |
| `--palette-plum` | `#37013A` | Carousel accent-panel background, gallery index 4 |

Cycles with pink (`ACCENTS[0]`) via `ACCENTS[index % ACCENTS.length]`, exactly matching the
prototype's `ACCENTS` array (`Atelier Homepage.dc.html` lines 187-193). **Never** used for text,
content-area backgrounds, or any UI state outside the carousel's own accent panel — this is a
decorative-only palette, confirmed by both D-05 and the `colors-accent-destructive.card.html`
guideline specimen. With only 2 migrated galleries live this phase (D-12: Silos, Brume), only
`ACCENTS[0]` (pink) and `ACCENTS[1]` (purple) are reachable in production today — implement the
modulo-cycling generically anyway so it extends correctly once the remaining 4 galleries migrate
(RESEARCH.md Open Question 3), not hardcoded to "2 galleries, 2 accents."

**Link hover/focus states** (`colors-link-states.card.html`): default link = pink text; hover/active
= pink text + underline (`text-decoration-thickness: 2px` on hover, `text-underline-offset: 3px`);
`:focus-visible` = `outline: 2px solid var(--color-accent); outline-offset: 2px;` — a **single** pink
outline ring, not the double ink+pink ring from the superseded Phase 2 identity (that co-styled
ring existed only because Wild Strawberry text failed contrast on Dawn Pink; pink-on-white in this
identity does not have that problem, so the simpler single-ring focus style from the imported
design system applies as-is).

## Layout Notes (phase specifics, not covered by the template's fixed sections)

### Header (site-wide, D-04/D-11/D-13)
- Header and footer merge onto `--color-dominant` (white), delineated by a `1px solid var(--color-border)` hairline (bottom border on header, top border on footer) — no background-color difference between chrome and content.
- Real AJS wordmark PNG logo (`AJS_Brutalist_Black_Transparent.png`) replaces the text-only site title, header only (D-11 — footer and favicon explicitly out of scope, deferred).
- **Logo hover-invert (D-13):** two stacked `<img>` elements — default state shows `AJS_Brutalist_Black_Transparent.png`; on `:hover`/`:focus-within`, cross-fades (opacity, 150ms ease) to `AJS_Brutalist_White_Transparent.png`. Because the white PNG variant is only legible against a dark surface, and D-04 makes the header white-on-white, a small dark ink (`#1A1A1A`) backdrop chip fades in behind the logo simultaneously (logo bounding box + `var(--space-sm)` padding on all sides, `--radius-none` square corners). **This backdrop-chip mechanism is a default, not independently re-verified against the live site this session** (RESEARCH.md Open Question 1, Assumption A3) — before shipping, do a 30-second check of `atelierjacquelinesuzanne.fr`'s current header hover behavior and adjust the chip's size/color only if it visibly conflicts; the crossfade-between-two-PNGs mechanism itself is locked regardless.
- On the gallery-detail page's transparent/scrim header variant (D-07), the header already renders over a dark photo + gradient scrim — the white logo PNG variant is used as the default (not hover-triggered) in that context, matching the prototype's `Header` reference component's `transparent` prop (`logo-src="...White_Transparent.png"` when `transparent={true}`).

### Homepage — Carousel mode (default view, D-08/D-09)
- Full-viewport (`height: 100vh; min-height: 680px`) hero section. Full-bleed cover photo of the current gallery + a dark gradient scrim (`linear-gradient(to top, rgba(0,0,0,0.65) 0%, rgba(0,0,0,0.15) 45%, rgba(0,0,0,0) 70%)`) for text legibility (D-07's technique, reused here).
- Dark top bar (own gradient, `rgba(0,0,0,0.5)` → transparent) overlays: white logo PNG (left), nav links + carousel/grid toggle + LanguageSwitcher (right) — all rendered in the current accent color (`{{ accentBg }}`, i.e. pink by default, cycling per gallery) per the prototype, not a fixed white/ink color.
- Bottom-left: index label ("01 / 02", Label role, `rgba(255,255,255,0.75)`), current gallery title uppercased (Heading role, white), byline (Label role, `rgba(255,255,255,0.75)`).
- Bottom-right: accent panel, `width: min(560px, 52%)`, background = current `ACCENTS[index % 5]` entry, containing the fluid-Display "Atelier Jacqueline Suzanne" wordmark text, intro body copy, and the underlined CTA link.
- Prev/next arrow buttons (`‹`/`›` glyphs), vertically centered left/right, 44px tap target, `rgba(0,0,0,0.35)` background chip, white glyph color.
- **Auto-advance (D-09):** `setInterval`, advances `carouselIndex` every **6000ms** (RESEARCH.md Assumption A4 — no interval was specified in CONTEXT.md/the prototype; 6s is this document's explicit default, a one-line tuning value if it feels wrong in review). Paused on `mouseenter`/`focusin` of the carousel root, resumed on `mouseleave`/`focusout` — never paused permanently, matches D-09's explicit departure from the design system's own motion-neutral default guidance.
- Only galleries with real migrated images appear (D-12 — currently Silos, Brume; filter applied in `index.astro`'s frontmatter via `getGalleries()`, not a new query function).

### Homepage — Grid mode (toggle target, D-08)
- Same top bar (nav + toggle + switcher), but static-position (not absolute-over-hero) since there's no hero image beneath it — background `none`, ink-colored (not white) text/icons since it now sits directly on the white page background.
- Below the top bar: a full-width pink (`--color-accent`) intro band containing the same wordmark text + intro copy (no CTA link repeated here — the grid itself is the "discover galleries" affordance).
- 2-column grid (`grid-template-columns: repeat(2, 1fr); gap: 2px`) of gallery tiles, each: full-bleed cover photo + the same dark gradient scrim + gallery title (Display role, fixed 32px/600, white) anchored bottom-left. Clicking a tile navigates to that gallery's detail page.
- Toggle state persists only for the current page view (no localStorage/cookie persistence specified or needed — refreshing the homepage always returns to carousel mode, the default).

### Homepage — Mobile responsive (D-10, Claude's Discretion — resolved here)
- Breakpoint: **768px**, matching the single breakpoint already used site-wide (`BaseLayout.astro`, gallery grids).
- Below 768px: hero carousel stacks vertically — hero photo area shrinks to `min-height: 420px` (matching the existing gallery-detail hero's mobile-safe floor, RESEARCH.md/`02-UI-SPEC.md` precedent), accent panel becomes full-width and sits **below** the hero image in normal document flow (not absolutely positioned over it) rather than the desktop's bottom-right `52%`-width overlay. Bottom-left index/title/byline block moves to sit above the accent panel, still overlaid on the hero photo itself (not pushed into the stacked flow) since it's a caption for the image directly above the panel.
- Grid mode below 768px: single column (`grid-template-columns: 1fr`), same tile treatment, unchanged intro band (already full-width by design).
- Top bar nav (logo, nav links, toggle, switcher) does not need a hamburger/collapsed treatment — the existing nav item count (Accueil/Home, Galeries/Galleries, Carrousel/Grille toggle, FR|EN) is short enough to wrap onto a second line at narrow widths without a menu pattern, consistent with how `BaseLayout.astro`'s standard header already behaves (no existing hamburger pattern in this codebase to depart from).

### Homepage intro copy source (Claude's Discretion, RESEARCH.md Open Question 2 — resolved here)
- **Decision: hardcode the intro copy** (wordmark text, intro body, byline, CTA) as locale-conditional strings directly in `index.astro`/`en/index.astro`, matching the prototype word-for-word, rather than repurposing `siteSettings.welcomeHeading`/`welcomeBody`. Rationale: the prototype's copy is a fixed, art-directed pairing with the fluid wordmark treatment and the CTA link — not a freely-editable paragraph Romane would plausibly rewrite day-to-day (unlike gallery captions, which are already CMS-driven). This leaves `welcomeHeading`/`welcomeBody` unused after this phase (safe — `getSiteSettings()` already null-safes every field; no schema change needed). If Romane later asks to edit this copy herself, wiring it to those existing fields is a small follow-up, not a blocker now.

### Gallery-detail hero-scrim (D-07)
- Existing gallery-detail page header gets the same full-bleed-photo + dark-gradient-scrim + white-text treatment as the homepage carousel (`Atelier Homepage.dc.html` lines 130-141): `height: 70vh; min-height: 420px`, transparent Header variant (white logo PNG, white nav text) absolutely positioned over the scrim, "← Retour aux galeries"/"← Back to galleries" back-link, project title as an absolutely-positioned bottom-left `<h1>` (Display role, fixed 32px/600, white text). Artist statement and thumbnail grid below remain unchanged in structure (repaint only).

### Contact form (D-14/D-15)
- Markup refactors to use `Input`/`Textarea`/`Button` components in place of the hand-rolled `<input>`/`<textarea>`/`<button>` + inline `<style>` rules currently in `ContactForm.astro`.
- The `<script>` block's DOM queries (`data-role="submit-button"`, `[data-role="error-name"]`, etc.) must continue to resolve against whatever markup the new components render — either the components accept/forward a `data-role` prop, or the refactor hardcodes the attribute on this specific usage. Validation/honeypot logic (`src/lib/contact-form.ts`) is unchanged (V5 Input Validation, RESEARCH.md Security Domain — must not regress).
- Error style changes from red text to bold ink text + thicker ink border per D-03; this is a pure CSS/inline-style change, not a validation-logic change.
