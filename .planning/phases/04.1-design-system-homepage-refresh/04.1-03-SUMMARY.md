# Phase 04.1 Plan 03: Form/Feedback Components + ContactForm & Galleries-Listing Refactor Summary

**Four new presentational Astro components (Button, Input, Textarea, EmptyState) built from design-import references, then wired into ContactForm and the galleries listing to finish eliminating `--color-destructive` from the codebase.**

## Performance

- **Duration:** ~35 min
- **Completed:** 2026-07-09
- **Tasks:** 3 (all auto)

## Accomplishments
- Created `Button.astro`, `Input.astro`, `Textarea.astro`, `EmptyState.astro` — dependency-free, no React/JSX, no `client:*` directive, all reading Plan 01's token set
- Refactored `ContactForm.astro` onto the new Input/Textarea/Button primitives while preserving every `data-role` the validation `<script>` queries and leaving `src/lib/contact-form.ts`'s `isBlank`/`isValidEmail`/`isHoneypotTriggered` untouched
- Replaced the last `--color-destructive` reference (an inline `style.color` JS assignment in `renderSubmissionError()`) with a `.contact-form__status--error` class toggle, and added a `:has()`-based ink border rule for errored fields
- Adopted `EmptyState` on both galleries listing pages (FR/EN), reusing existing copy verbatim, and converted the listing's Display heading from the superseded fluid `clamp()` treatment to the fixed 32px Display role
- Verified `grep -rn "color-destructive" src/` returns zero matches — the token is now fully eliminated site-wide

## Files Created/Modified
- `src/components/Button.astro` - Primary/secondary variant button, attribute-forwarding (`...rest`) so `data-role` reaches the rendered `<button>`
- `src/components/Input.astro` - Labelled text input, ink-only error border + `role="alert"` error text
- `src/components/Textarea.astro` - Same convention as Input, `rows` default 5, `resize: vertical`
- `src/components/EmptyState.astro` - Heading/body block, copy fully driven by props (no hardcoded FR-only default)
- `src/components/ContactForm.astro` - Now imports/uses Input, Textarea, Button; `renderSubmissionError()`/`renderSuccess()` toggle a CSS class instead of setting inline `style.color`; `<style>` block's error/status rules use `var(--color-ink)` + `var(--weight-semibold)`, no red
- `src/pages/galleries/index.astro` / `src/pages/en/galleries/index.astro` - Empty-state branch now renders `<EmptyState heading=... body=... />`; Display `h1` uses `var(--text-display-*)` tokens instead of `clamp(2.5rem, 12cqi, 6.5rem)` + `font-weight: 900`

## Decisions Made
- Kept the per-field error `<p role="alert">` elements as ContactForm-owned sibling markup (not passed through Input/Textarea's `error` prop) since the validation `<script>` needs to set their text imperatively at submit time — a static build-time prop can't represent that.
- Used a CSS `:has()` selector (`.contact-form__field:has(.contact-form__error:not(:empty)) :global(.field__input)`) to apply the ink-only 2px error border to Input/Textarea's own internal classes from ContactForm's scoped stylesheet, rather than adding a second JS-driven class toggle for something CSS can already express declaratively. `:has()` has broad support in current evergreen browsers.
- This worktree's `node_modules` was completely absent (not just missing the Wave-1-added `@fontsource/archivo-black` package) — ran `npm install` and copied the main repo's gitignored `.env` in before the first build, mirroring Plan 01's Task 2 deviation.
