## Spacing Scale

Existing site-wide tokens (`src/layouts/BaseLayout.astro` `:root`) — reuse as-is, no new spacing tokens needed for this phase:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding |
| sm | 8px | Compact element spacing |
| md | 16px | Default element spacing |
| lg | 24px | Section padding |
| xl | 32px | Layout gaps |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing |

Exceptions for this phase:

- **Blur placeholder image dimensions (D-01, locked):** `width(24).blur(50)` — a Sanity CDN image-processing parameter pair, not an 8-point spacing token. 24px is deliberately far below any realistic source photo's smallest dimension (Sanity scales small sources *up* to match requested width unless `.fit('max')` is set — see `09-RESEARCH.md` Pitfall 2 — irrelevant at this size, but do not raise the width without re-checking that caveat). Verify live per CONTEXT.md's own "Verify live" note: large/high-contrast enough to read as a recognizable color/shape preview for this specific set of photos (Silos, Brume, etc. — documentary/architectural, sometimes high-contrast), not a solid blob.
- **Crossfade transition duration (D-06, locked): ~260ms ease** (not a spacing value, but the phase's one new timing constant — recorded here since the template has no dedicated Timing section). Matches the existing toggle-morph convention (`.home-toggle__morph`'s `transition: column-gap 280ms ease`, `HomeCarousel.astro:956`) — quick and subtle, reads as "the photo just finished loading," not a slow/cinematic reveal. Applies identically to the hero crossfade and the grid-tile crossfade (D-03: same visual treatment, both surfaces).
- No new `--tap-target-min` or box-geometry exceptions — this phase touches no clickable control's hit area.

## Layout & Interaction Notes

*(Project convention — see Phase 6/7/8's UI-SPEC "Layout & Interaction Notes"; this section is the canonical source for this phase's visual mechanism, since HOME-09 is entirely a loading-behavior/interaction contract with no new spacing/typography/color tokens.)*

### Mechanism: two-layer opacity crossfade (Claude's Discretion, locked by this contract)

Per `09-RESEARCH.md`'s Architecture Patterns (Pattern 2/3) and Anti-Patterns, the executor MUST implement blur-up as **two stacked `<img>` elements**, not a single `<img>` with an animated `filter: blur()`:

1. A blurred placeholder `<img>` (Sanity CDN URL, D-01: `.width(24).blur(50).auto('format').url()`), `alt=""`, `aria-hidden="true"`, absolutely positioned to fill its container (`position: absolute; inset: 0; width/height: 100%; object-fit: cover`) — same box as the sharp image it sits beneath.
2. The sharp `<img>` (existing `heroSrc`/`gridSrc`), stacked on top via the same absolute-positioning box, starting at `opacity: 0`.
3. `transition: opacity 260ms ease` on the sharp layer only (D-06). A `load` listener (or `img.complete` check for the already-cached case — reuses the exact defensive pattern already in `render()`'s `syncWordmarkAlignment` hookup) adds an `.is-loaded` class that animates the sharp layer's `opacity` to `1`.

This is a hard requirement, not a style preference: swapping `src` on a single `<img>` mid-blur-animation produces a visible pop (wrong-resolution frame + jump-cut) — the two-layer approach never shows an unblurred wrong-resolution frame (see `09-RESEARCH.md` Alternatives Considered / Anti-Patterns). Do not substitute a single-element `filter: blur()` animation.

### Hero (every swap — D-02, D-04, D-05)

- **Priority:** the hero `<img>` (`[data-role="hero-image"]`) gets `fetchpriority="high"` unconditionally, server-rendered in the initial HTML (D-04). No `loading="lazy"` on this element, ever.
- **Placement:** the new placeholder `<img data-role="hero-image-placeholder">` is a sibling of the existing `[data-role="hero-image"]`, both inside `.home-hero__photo` (which already establishes `position: relative` and hosts other absolutely-positioned children — `.home-hero__scrim`, `.home-hero__caption`). Placeholder renders first in DOM order (paints beneath), sharp image second (paints above, `z-index` follows DOM order since neither needs an explicit stacking value beyond what already exists).
- **Every swap, not just first paint (D-02):** `render()` (the existing client-side function that already mutates `heroImg.src`/`alt` on every auto-advance/prev/next/toggle/progress-dash-click tick) is extended to: (1) set the placeholder's `src` to the new gallery's blurred URL, (2) **remove** the sharp image's `.is-loaded` class *before* reassigning its `src` (required — CSS classes persist across `src` reassignment, so skipping this step leaves the previous photo's opacity:1 state visually stuck over the new, still-decoding photo — see `09-RESEARCH.md` Pitfall 3), (3) reassign the sharp image's `src`, (4) re-attach the `load`/`complete` fade-in check.
- **Prefetch (D-05):** at the end of every `render()` call, prefetch `galleries[(carouselIndex + 1) % galleries.length].heroSrc` via a bare `new Image()` (no `<link rel="prefetch">` — simpler to trigger from inside existing client JS, no `<head>` DOM management). This means the blur-up moment is the *exception* (only visible on true first paint or a cold cache), not something visible on every single swap — by the time the next auto-advance/prev/next/toggle fires (~6s later on auto-advance), the sharp photo is typically already warm in the browser's HTTP cache and the crossfade resolves near-instantly.
- **Do not tie `--wordmark-photo` to the new load listener (Pitfall 4):** `render()` already sets `root.style.setProperty('--wordmark-photo', 'url(' + gallery.heroSrc + ')')` synchronously and unconditionally, independent of the sharp `<img>`'s load state — this is correct existing behavior (the wordmark cutout uses its own CSS `background-image`, not the `<img>` element) and MUST stay exactly where it is. Do not move it inside the new `load`/`is-loaded` logic — doing so would regress the wordmark to lag one gallery behind the caption/title text.

### Grid tiles (lazy, as they scroll into view — D-03)

- Grid tiles are server-rendered once (`src/pages/index.astro` → `HomeCarousel` props), not re-rendered by `render()` — so this is primarily a server-rendered markup + CSS concern, with one small standalone client-side listener (not `render()`) driving the fade.
- Each `.home-grid__tile` gets the same placeholder-under-sharp structure: a new `<img class="home-grid__tile-img home-grid__tile-img-placeholder" src={gallery.blurSrc} alt="" aria-hidden="true">` stacked beneath the existing `<img class="home-grid__tile-img home-grid__tile-img--sharp" ...>`.
- **The existing `loading="lazy" decoding="async"` attributes on the sharp grid-tile `<img>` MUST be kept, unchanged (D-03's explicit instruction)** — this phase adds the blur-up crossfade *on top of* lazy-loading, it does not replace or remove it. A regression test should assert `loading="lazy"` is still present after this phase ships.
- `.home-grid__tile-img` (the shared base class) currently sits in normal flow (`width/height:100%; object-fit:cover; display:block`); stacking the placeholder requires adding `position: absolute; inset: 0;` to both the placeholder and sharp variants. This is additive to `.home-grid__tile`'s existing `position: relative` (already established for `.home-grid__tile-scrim`/`-copy`), not a new stacking pattern.
- A one-time client-side setup (added alongside the other `querySelectorAll` calls in `HomeCarousel.astro`'s existing `<script>` block, not inside `render()`) attaches the same `load`/`complete` fade-in check to every `.home-grid__tile-img--sharp` element.

### View Transitions coexistence (must not regress HOME-06/HOME-07/HOME-08)

This phase's crossfade elements sit on the exact same elements the already-shipped View Transitions morph (`ajs-hero-morph`/`ajs-accent-panel`/`ajs-header`, Phase 7 / quick-tasks `260713-jfz`/`260713-kit`) touches. The blur-up mechanism MUST coexist with, not fight, that existing wiring:

- **Never assign a `view-transition-name` to the new placeholder `<img>` or any new crossfade wrapper element.** It must ride along as part of `.home-hero__photo`'s (or the grid tile's) existing snapshot, unnamed — giving it its own name would create a second named group fighting the existing shared-element morph (see `09-RESEARCH.md` Anti-Patterns).
- **Known, accepted cosmetic edge case (not a blocker, do not over-fix):** during the carousel↔grid toggle click specifically (the only interaction that calls `document.startViewTransition()`), if a hero or grid-tile crossfade is mid-animation at the exact moment the toggle fires, that live opacity transition is briefly upstaged by the View Transition overlay's own snapshot-based cross-fade and may appear to "jump" to its final state once the overlay tears down, rather than visibly dissolving. Per the `07-CONTEXT.md` D-10/D-12 precedent (an unconfirmed hypothesis was NOT over-fixed with defensive gating), do not add `pointer:fine`/viewport gating or disable the crossfade during transitions to pre-empt this — it is a narrow, low-severity, rare-coincidence artifact, not a functional break. Verify visually during implementation (click toggle immediately after a fresh gallery swap); only act further if a real, reproducible artifact is observed.

### Data flow (build-time → client, new field)

- `src/lib/image.ts`: add `blurPlaceholderUrl(img, width = 24)` alongside `thumbnailUrl()`/`fullSizeUrl()`, using the same builder singleton (`.width(24).blur(50).auto('format').url()`, D-01).
- `src/pages/index.astro` / `src/pages/en/index.astro`: compute `blurSrc: blurPlaceholderUrl(cover)` in the existing `galleries.map()`, exactly where `heroSrc`/`gridSrc` are already computed, and pass it through to `HomeCarousel` props.
- `HomeCarousel.astro`: add `blurSrc: string` to the `GalleryEntry` interface; thread it through the hidden `<ul data-role="home-carousel-data">` data island as a new `data-blur-src` attribute (mirrors `data-hero-src` exactly); parse it client-side in the existing `Array.from(dataEl.querySelectorAll('li')).map(...)` block.
- **Security boundary preserved (unchanged from Phase 04.1's threat model, T-04.1-04-ID):** `blurPlaceholderUrl()` is called only from Astro frontmatter (`index.astro`/`en/index.astro`), never from `HomeCarousel.astro`'s client `<script>`. The client script only ever reads the pre-computed `data-blur-src` attribute off the static data island — no new import of `src/lib/sanity.ts` is introduced, so `SANITY_API_READ_TOKEN` never reaches the browser.

### Page-shell rendering (success criterion #1 — no new work required)

There is no existing blocking full-screen loader in `HomeCarousel.astro` today — the header/nav/toggle render synchronously in the initial server-rendered HTML with no async gate in front of them. HOME-09's "no blocking full-screen loader" criterion is satisfied by **not introducing one** in this phase (e.g. no wrapping the whole `<section class="home">` in a loading-state conditional, no skeleton screen that blocks chrome). This is a negative/regression contract, not new UI to design.
