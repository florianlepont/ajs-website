> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none — plain Astro components, vanilla CSS custom properties (no React/Vue/Svelte island anywhere in the codebase, no `components.json`, no Tailwind/PostCSS config found) |
| Icon library | none — hand-authored inline `<svg>`/CSS-grid glyphs (dependency-free, existing codebase convention: `.home-toggle__morph`'s 6-cell grid icon, the Instagram camera glyph, and this phase's new globe glyph all follow the same rule) |
| Font | `--font-display: 'Unbounded', var(--font-sans)` (self-hosted, `@fontsource/unbounded/900.css`, 900 weight only) for Display/Heading roles; `--font-sans` (system-ui stack) for Body/Label roles |

**shadcn gate note:** No `components.json` and no React/Next.js/Vite dependency in `package.json` (Astro 7.0.6, zero framework islands). This phase is a pure component-extraction/consolidation refactor of the existing hand-rolled token system (`:root` custom properties in `src/layouts/BaseLayout.astro`) — introducing shadcn here would contradict the phase's own goal (unify two implementations onto one canonical, already-proven pattern) and every prior homepage phase's determination (Phases 04.1/04.2/04.3/6/7/9). Skipped per the gate's own scoping (tech stack is not React/Next.js/Vite), not asked.

**This phase introduces zero new design tokens.** It is a structural consolidation (`<SiteHeader>` extraction, HOME-10) plus a copy/markup simplification (`LanguageSwitcher`, I18N-04) built entirely from spacing/typography/color values that already exist in `BaseLayout.astro`'s `:root` block. The sections below document how those existing values apply to the two things that materially change: the unified header's class contract, and the simplified switcher's new icon+label composition.

## Spacing Scale

Existing site-wide tokens (`src/layouts/BaseLayout.astro` `:root`) — reuse as-is, no new tokens needed:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding |
| sm | 8px | Compact element spacing |
| md | 16px | Default element spacing |
| lg | 24px | Section padding |
| xl | 32px | Layout gaps |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing |

Exceptions for this phase:

- **`--tap-target-min` (44px)** — already an established token (`BaseLayout.astro`, `HomeCarousel.astro`, `LanguageSwitcher.astro`). Reused, not new. `<SiteHeader>`'s Instagram link, the mode-toggle's outer hit-box, and the simplified switcher's single remaining link must all keep this floor — none of that changes in this phase, it just now applies identically wherever `<SiteHeader>` renders.
- **Switcher icon+label gap: `var(--space-xs)` (4px), new declaration on `.switcher-link`.** Today `.language-switcher` puts its 4px gap on the *nav* (between two `<a>` children and a separator); after I18N-04 there is only one `<a>` child, so that nav-level gap becomes moot. The new gap lives one level down, between the globe `<svg>` and the visible "EN"/"FR" text node inside the single `<a>` (`display:inline-flex; align-items:center; gap: var(--space-xs);`) — same token, relocated to where it's now needed.
- **Mobile pixel-budget re-measurement is REQUIRED on both sides of this consolidation, not just the homepage (RESEARCH.md Pitfall 1/2):**
  - `BaseLayout.astro`'s header currently has **zero** `@media (max-width: 767px)` rules — it has never needed one at its historical 4-item row (logo, About, Contact, switcher `"FR | EN"`). Once `<SiteHeader>` adds Instagram as a 3rd nav link (D-03) everywhere, About/Contact/gallery-detail inherit a mobile-fit risk they've never had before. **Port `HomeCarousel.astro`'s existing, live-tested `@media (max-width: 767px)` block forward into `<SiteHeader>`'s own stylesheet** (translated to the unified class names below), rather than leaving BaseLayout's untested mobile behavior as the baseline or writing new mobile CSS from scratch.
  - The homepage's own 393px-viewport pixel budget changes on **both** sides of the equation this phase touches: the switcher shrinks substantially (two links + `"|"` separator → one link + a 16px icon + one language code), while Instagram was already counted in Phase 7's budget (unaffected). Do not assume the new switcher's rendered width — measure it live (`getBoundingClientRect()` / Playwright `boundingBox()`) at 393px and re-verify the full header still fits on one row, per the same "live-measure-and-adjust" methodology as Phase 7 (Claude's Discretion, CONTEXT.md).
  - New floor to also verify: About/Contact/gallery-detail at ~375–428px with the now-present Instagram link and the now-shrunk switcher — this class of regression (`<SiteHeader>` overflow/wrap on non-homepage pages) has no existing automated test; treat it as a required live/Playwright spot-check, not optional polish.

## Color

Existing site-wide tokens — **unchanged by this phase**:

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#FFFFFF` (`--color-dominant` / `--gray-0`) | Page background; `.site-header--solid` background (About/Contact) |
| Secondary (30%) | `#FFFFFF` (`--color-secondary`, merged into Dominant per Phase 4.1 — header/footer are white-on-white) | Same surfaces as Dominant |
| Accent (10%) | `--color-accent` (`#FF3B94` pink-600) | Default link color (`a { color: var(--color-accent) }`), focus rings, and — pre-existing, untouched by this phase — the per-gallery cycling accent (carousel intro panel, grid hero tile, gallery-tile hover scrim) |
| Destructive | N/A | No destructive actions in this phase |

**This phase introduces no new color token.** What changes is *which selectors* the existing solid/transparent variant colors apply through, now that one component serves every page:

- **`.site-header--solid` (About/Contact, and grid-mode homepage once re-skinned — D-02):** white background, `--border-hairline` bottom border, `.nav-link`/logo render ink (`--color-ink`). Unchanged from `BaseLayout.astro`'s current solid header.
- **`.site-header--transparent` (gallery-detail, and carousel-mode homepage once fixed-prop-rendered — D-02):** absolutely-positioned dark gradient scrim, `.nav-link` forced to `#FFFFFF`. Unchanged from `BaseLayout.astro`'s current transparent header.
- **`.switcher-link` default color stays `var(--color-accent)` (pink) everywhere `<SiteHeader>` renders it** — this is `LanguageSwitcher.astro`'s own hardcoded color, unaffected by the header's solid/transparent variant class (confirmed: today's `BaseLayout.astro` transparent variant, used on gallery-detail, already leaves the switcher pink against its dark scrim — a pre-existing, already-shipped combination this phase does not need to change). **Only the homepage's own two `[data-display-mode]` states override `.switcher-link`'s color** (white in carousel mode, ink in grid mode) — this override must be ported from its current selector (`.home[data-display-mode='...'] .home-header :global(.switcher-link)`) to the unified name (`.home[data-display-mode='...'] .site-header :global(.switcher-link)`), and it **stays scoped inside `HomeCarousel.astro`'s own stylesheet**, not moved into `<SiteHeader>`'s global styles — it is homepage-specific re-skinning, not a property of the shared component itself (RESEARCH.md Pattern 1/3).
- **Globe icon (new):** `stroke="currentColor"` — inherits whatever color `.switcher-link` currently has (pink by default, white/ink on the homepage per the override above). No separate icon color token.
- **Instagram icon (unchanged, D-03 only relocates it):** `stroke="currentColor"` inheriting `.nav-link`'s color, which already flips ink/white per `.site-header--solid`/`--transparent` — same mechanism, now reachable from every page instead of only the homepage.

Accent reserved for: unchanged — link color, focus rings, and the per-gallery cycling accent (carousel intro panel background, grid hero-tile background, gallery-tile hover scrim). **Not** the nav links, Instagram icon, globe icon, mode-toggle, or switcher link — all of those stay ink/white/pink per the existing header-chrome contract described above, not the cycling gallery accent.

## Layout & Interaction Notes

*(Project convention — see Phase 6/7's `06-UI-SPEC.md`/`07-UI-SPEC.md` "Layout & Interaction Notes"; this section is the canonical source for this phase's structural contract.)*

### `<SiteHeader>` component contract (HOME-10, D-01–D-04)

- **Fixed-prop rendering, never client-side re-render (RESEARCH.md Pattern 1 — load-bearing):** `<SiteHeader>` is a plain `.astro` component with zero hydration. It renders to static HTML exactly once, with a fixed `variant` prop. There is no code path — now or ever, in this stack — where JS "changes" `<SiteHeader>`'s variant after the page loads. The homepage's carousel↔grid visual swap works exactly like every other display-mode-reactive element already in `HomeCarousel.astro` (logo hover-swap, wordmark sizing, toggle-icon morph): a single `data-display-mode` attribute mutated on the `.home` root by a plain `<script>`, with CSS attribute-selectors doing all the re-skinning. `<SiteHeader variant="transparent">` renders once on the homepage (matching the initial carousel state); grid mode is achieved by a page-scoped CSS override that re-skins the already-rendered transparent markup to look solid — not by swapping the prop.
- **Props (2-value variant union, RESEARCH.md Open Question 2 resolution):** `variant: 'solid' | 'transparent'` — **not** a 3-value union with `'none'`. The `'none'`-suppression behavior stays entirely in `BaseLayout.astro`'s own conditional (`{headerVariant !== 'none' && <SiteHeader variant={headerVariant} />}`), exactly mirroring today's conditional around the inline `<header>`. `<SiteHeader>` itself only ever needs to know how to paint a header, never whether to paint one at all.
- **Named slot for the mode-toggle only (D-04):** `<slot name="extra" />`, positioned between the nav and the `<LanguageSwitcher />` — matching the current DOM order (`nav → toggle → switcher`) and D-04's grouping logic (the toggle is a display control, grouped near the switcher, not the "leave the page" About/Contact/Instagram links). This is the **only** content ever passed through this slot; every other page renders `<SiteHeader>` with nothing in it.
- **Class/attribute rename map (D-05, this phase's concrete resolution of CONTEXT.md's Claude's Discretion):**

  | Retired (`.home-*`, homepage-only) | Unified (renders everywhere via `<SiteHeader>`) |
  |---|---|
  | `<header class="home-header" data-role="home-header">` | `<header class:list={['chrome-band','site-header', \`site-header--${variant}\`]} data-role="site-header">` |
  | `.home-nav` | `.site-nav` |
  | `.home-nav-link` | `.nav-link` |
  | `.home-logo` | `.logo-mark` |
  | `.home-logo__chip` | `.logo-mark__chip` |
  | `.home-logo__img` / `--default` / `--hover` | `.logo-mark__img` / `--default` / `--hover` |
  | `[data-role="home-header"]` | `[data-role="site-header"]` |

  `.home-toggle`, `.home-toggle__box`, `.home-toggle__morph*`, and `[data-role="mode-toggle"]` are **unaffected** — D-05 explicitly excludes the mode-toggle from this rename (it names an inherently homepage-only interactive element, not a header-duplication artifact).
- **CSS scoping: `is:global`, not Astro's default scoped `<style>` (RESEARCH.md Pattern 3).** `BaseLayout.astro`'s current header CSS already lives in `<style is:global>`. `<SiteHeader>`'s own stylesheet must keep this convention — an auto-scoped `<style>` would force every cross-file override (the homepage's `[data-display-mode]` re-skin rules, the mobile `@media` port, `LanguageSwitcher`'s external color overrides) into `:global(...)` wrappers for no benefit, and would diverge from the one working pattern this whole component is extracted from.
- **Cross-page structural identity is itself part of the visual contract, not just a copy contract:** the same nav-link count/order (Logo, About, Contact, Instagram, optional slot, switcher) must render on `/`, `/about/`, `/contact/`, and every `/galleries/[slug]/` — this is what "visually identical by construction" (HOME-10 success criterion #1) actually means at the DOM level, and is the gap RESEARCH.md flags as untested by any existing e2e assertion (Wave 0 gap).

### Simplified `<LanguageSwitcher>` (I18N-04, D-07–D-11)

- **One link, not two — no separator.** `getSwitcherHref` is called once, for the non-current locale only (D-08). The `<span class="switcher-separator">|</span>` and the current-language `<a class="switcher-link is-current">` both disappear entirely, along with the `.is-current`/`.switcher-separator` CSS rules that only existed to support them.
- **Link composition, left to right:** globe `<svg aria-hidden="true">` (16×16, `viewBox="0 0 24 24"`, `stroke="currentColor"`, `stroke-width="1.6"`, a circle + one vertical-axis ellipse + one horizontal meridian line — any standard, recognizable globe glyph is acceptable per CONTEXT.md's Claude's Discretion; verify legibility live at this rendered size, same precedent as Phase 7's Instagram icon), then the visible language-code text node, then the sr-only hint span. `.switcher-link` becomes `display:inline-flex; align-items:center; gap: var(--space-xs);` to lay these three children out (see Spacing Scale exception above).
- **Navigation/cookie mechanism is 100% unchanged (D-10):** the existing inline `<script>` still writes `ajs_locale` via `document.cookie` on click, scoped to the same base-aware `path`; the click still lands on `getSwitcherHref(Astro.url.pathname, targetLocale)`'s computed URL. Only the DOM/visual presentation changes (one link + icon instead of two links + separator).
- **Verify against the three dependent test files, not just the two CONTEXT.md names:** `tests/e2e/i18n.spec.ts` and `homepage.spec.ts` both need direct rewrites (the literal `"FR | EN"` text assertion is provably false after this ships); `tests/e2e/legal.spec.ts` also depends on switcher click behavior via `getByRole('link', { name: 'EN' | 'FR' })` and needs **no code change** as long as the accessible-name contract above (visible label never `aria-hidden`) is followed — but it must still be run as a regression check (RESEARCH.md Pitfall 4).
