# Phase 3 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Carry-Forward Note

This phase introduces **no new visual identity** — it reuses the site-wide design system locked in `02-UI-SPEC.md` (Dawn Pink / Woodsmoke / Wild Strawberry, Delight font, 4-role/2-weight typography, 8-point spacing scale) verbatim, per that document's own framing ("Applies site-wide", single point of change in `BaseLayout.astro`). Nothing in `03-CONTEXT.md` requests a new color, font, or spacing scale. This document therefore:

1. Confirms the existing tokens apply unchanged (no re-litigation).
2. Declares the **first live use** of the `--color-destructive` token (`#dc2626`, forward-declared in Phase 2 but unused until now) for contact-form validation errors.
3. Declares role assignments, copy, and interaction states specific to the About page and the Contact form — the only genuinely new surface this phase adds.

Sources used to pre-populate this contract: `03-CONTEXT.md` `## Decisions` (D-01 through D-12, all locked), `src/layouts/BaseLayout.astro` (existing tokens), `02-UI-SPEC.md` (established role/usage precedent). No `RESEARCH.md` exists yet for this phase (runs after this document, during `/gsd:plan-phase`) — the contact form's backend delivery mechanism (D-08) is explicitly out of scope here and does not change any of the visual/interaction contract below.

## Spacing Scale

Declared values (must be multiples of 4) — unchanged from Phase 1/2:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding |
| sm | 8px | Compact element spacing (e.g. field label-to-input gap) |
| md | 16px | Default element spacing (subheading-to-paragraph gap, page horizontal padding) |
| lg | 24px | Section padding |
| xl | 32px | Layout gaps (form field-to-field vertical gap) |
| 2xl | 48px | Major section breaks (bio-to-atelier section break on About page) |
| 3xl | 64px | Page-level spacing |

Exceptions:
- **44px minimum tap target** on all contact-form interactive elements (name input, email input, submit button) — same WCAG 2.5.5 exception already established for `.switcher-link` (Phase 1) and lightbox controls (Phase 2). Applied via padding, not by inflating font-size.
- **Textarea minimum height**: 120px (message field) — a usability floor for a multi-line field, not a tap-target rule.
- No other exceptions.

## Layout Notes (Phase 3 specifics, not covered by the template's fixed sections)

### About Page

- Single page, flowing sections top-to-bottom — no tabs, no sub-navigation, no visually-boxed panels between sections (D-02, locked). Structure:
  1. `<h1>` — Display role, plain, no panel (see Typography role table above).
  2. Bio section — Body-role paragraph(s), migrated verbatim from the current Myportfolio site (D-01), nothing dropped (D-04).
  3. Atelier/practice subheading — Heading role, e.g. **FR:** "Atelier & pratique" · **EN:** "Studio & practice".
  4. Atelier/practice paragraph(s) — Body role, existing content migrated where available, lightly expanded only where the current site is incomplete (D-05).
  5. Placeholder paragraph for medium/technique specifics not yet confirmed by Romane (D-06) — **do not infer or guess** (e.g. do not assert analog/film/darkroom process from photo grain alone). Rendered as a normal Body-role paragraph with `font-style: italic` as the sole visual marker (no box, border, or background — consistent with D-02's "no boxed panels"), reading as a natural, visitor-facing "more soon" note rather than a raw TODO:
     - **FR:** "*Précisions sur le médium et la technique à venir — en attente de confirmation avec l'artiste.*"
     - **EN:** "*Details on medium and technique coming soon — pending confirmation with the artist.*"
- No portrait/photo of Romane anywhere on this page (D-03, locked) — text/practice-focused only.
- Vertical rhythm: `var(--space-2xl)` (48px) between the bio block and the atelier/practice block (matches the "major section breaks" usage already established in `02-UI-SPEC.md`); `var(--space-md)` (16px) between a subheading and its immediately-following paragraph.
- Max content width: `640px`, centered — reuses the exact reading-width precedent already established by the homepage `.welcome` block and the gallery-detail artist statement (`02-UI-SPEC.md` Layout Notes), not a new value.
- Content source: whether this copy is Sanity-editable or hardcoded in the `.astro` page is a planning/research decision (not a visual one) — `03-CONTEXT.md` code_context defaults to hardcoded content unless research surfaces a reason Romane needs to self-edit her bio. Either way, the rendered typography/spacing/copy contract above is identical.

### Contact Page

- Single page: `<h1>` (Display role, plain, no panel) → optional one-line lead-in (Body role, e.g. **FR:** "Une question, une envie de collaboration ? Écrivez-moi." · **EN:** "A question, or an idea for a collaboration? Get in touch.") → form.
- Form fields, in this order, all required (D-10, D-11 — locked, no optional fields, no subject/reason field):
  1. Name — `<input type="text">`
  2. Email — `<input type="email">`
  3. Message — `<textarea>`, `min-height: 120px`
  4. Honeypot (hidden, not visitor-facing — see below)
  5. Submit button
- Field labels are always visible above each input (never placeholder-only text, an accessibility anti-pattern) — Label role:
  - **FR:** "Nom" · **EN:** "Name"
  - **FR:** "E-mail" · **EN:** "Email"
  - **FR:** "Message" · **EN:** "Message"
- Field/button sizing: `min-height: 44px` on the name input, email input, and submit button (WCAG 2.5.5 tap-target floor, same exception pattern as `.switcher-link` and the lightbox controls). Vertical gap between fields: `var(--space-xl)` (32px). Label-to-input gap: `var(--space-sm)` (8px).
- **Honeypot field (CONT-02):** a decoy text input named `website` (a common bot-autofill target), visually and programmatically hidden from real visitors: `position: absolute; left: -9999px;` (not `display:none`/`visibility:hidden`, which some bots specifically detect and skip filling), plus `tabindex="-1"`, `aria-hidden="true"`, `autocomplete="off"`. Its label is hidden the same way — never shown, never announced. On submit, if this field is non-empty, treat the submission as spam: silently discard it and still render the same success confirmation shown to real senders (never reveal the detection mechanism to whoever/whatever submitted it). This defines the interaction contract only; the send/skip technical implementation is a research/planning concern per D-08.
- Focus states: reuse the existing site-wide double-ring `:focus-visible` pattern from `BaseLayout.astro` (`outline: 2px solid var(--color-ink); outline-offset: 2px; box-shadow: 0 0 0 4px var(--color-accent);`) on all three visible fields and the submit button — no new focus treatment invented for this phase.
- Submit button: Wild Strawberry background, Woodsmoke text, Heading-role weight (900), no border — the exact "Buttons/CTAs" convention forward-declared (but unused) in `02-UI-SPEC.md`; this is the first phase to actually ship one.

### Form Interaction States

- **Field-level validation** (client-side, checked on blur and on submit attempt) — Label role (14px/300), `color: var(--color-destructive)`, rendered directly below the affected field inside a `role="alert"` container so assistive tech announces it immediately:
  - Name empty — **FR:** "Merci d'indiquer votre nom." · **EN:** "Please enter your name."
  - Email empty — **FR:** "Merci d'indiquer votre e-mail." · **EN:** "Please enter your email."
  - Email malformed — **FR:** "Merci d'indiquer une adresse e-mail valide." · **EN:** "Please enter a valid email address."
  - Message empty — **FR:** "Merci d'indiquer votre message." · **EN:** "Please enter a message."
- **Submitting/loading state** — submit button becomes disabled (`opacity: 0.6; cursor: not-allowed;`) and its label swaps to **FR:** "Envoi en cours…" · **EN:** "Sending…" — text-only, no spinner icon (avoids introducing new iconography for a single transient state).
- **Success (D-09, locked — inline, no redirect)** — on successful submit, an `aria-live="polite"` region (inserted above or below the form; exact placement is an executor implementation choice) displays, Body role, Woodsmoke ink:
  - **FR:** "Merci, votre message a bien été envoyé. Je vous répondrai bientôt."
  - **EN:** "Thank you, your message has been sent. I'll get back to you soon."
- **Form-level submission error** (network/backend failure, distinct from field validation) — same `aria-live="polite"` region, Body role, `color: var(--color-destructive)` — copy given in the Copywriting Contract above (exact mailbox alias confirmed during planning).
- Max content width for the whole Contact page: `640px`, centered — same reading-width precedent as the About page and homepage.
