Fixes audit findings #1 (dark mode) and #2 (inconsistent gray rows), which are coupled because both hinge on the `--dashboard-page-background` / `--dashboard-surface-background` tokens.

DARK MODE (#1): In `EditorialDashboard.css`, delete the two hardcoded light-only hex custom properties defined on `.editorial-dashboard__page` (the page-background and surface-background literals, lines 2-3). Re-derive both from Sanity UI's own runtime, scheme-aware Card tokens — the same family the rest of this file already consumes (`var(--card-border-color)`, `var(--card-muted-fg-color)`, `var(--card-focus-ring-*)`) and that `DocumentChecklist.tsx`/`MediaLibrary.tsx` rely on. Concretely: the `.editorial-dashboard__surface` background should resolve to the Card's own scheme background — prefer `var(--card-bg-color)` (Sanity UI sets this on every Card, including the three `tone="default"` surface Cards that carry this class), or drop the CSS `background-color` override entirely so the Card renders its native tone-driven background. For the `.editorial-dashboard__page` outer div (a plain `<div>`, not a Card, so it does not itself receive `--card-bg-color`), give it a scheme-aware canvas: EITHER derive from an inherited/muted Sanity token that stays a light tint in light mode and a dark canvas in dark mode, OR add a `[data-scheme="dark"]` override block (Sanity UI writes `data-scheme` on a Studio root ancestor — confirmed present in `@sanity/ui` dist) that supplies the dark canvas value while a default rule keeps the light tint. Do not reintroduce a light-only hex that has no dark counterpart. Because `.editorial-dashboard__row-link:hover ... { background-color: var(--dashboard-page-background) }` reuses the page token for the hover state, keeping that token scheme-aware automatically keeps hover correct in both schemes — verify the hover shade still reads as a subtle tint, not white-on-white or a harsh block, in dark mode.

GRAY ROW (#2 — requires live inspection, cannot be validated headlessly): Form the strongest hypothesis from code first, apply the most likely fix, then confirm live in Task 3. Prime suspects to check, in order: (a) the `recommended` priority group's header Card uses `tone="default"` (see `buildAttentionGroups`), which Sanity UI renders as a neutral/gray card while the other groups' `critical`/`caution`/`primary` tones render tinted — a non-technical reporter may describe this per-group header difference as "gray background on some rows"; (b) the shared `:hover` rule painting the hovered row with `--dashboard-page-background` (a captured/resting pointer makes one row look permanently gray); (c) `tone="transparent"` `ContentRow` Cards inheriting a different resolved `--card-bg-color` depending on nesting once the forced-white surface override from #1 is removed. Apply the fix your hypothesis best supports (e.g. give the row/group backgrounds a single consistent scheme-aware token so every row in a group matches). Do NOT silently skip this — if code review cannot isolate the cause with confidence, still apply your best hypothesis and defer confirmation to the Task 3 checkpoint, where the user supplies live DevTools inspector output for a gray vs. non-gray row.

Do NOT touch the unauthenticated GitHub fetch, its rate-limit handling, or `sanity.config.ts` theme config — all out of scope.

<task type="auto">
  <name>Task 2: Fix error copy, exclude preparation drafts from urgent bucket, add heading semantics, and remove dead code</name>
  <files>sanity/editorial/EditorialDashboard.tsx, sanity/editorial/deployment.ts</files>
  <action>
Fixes audit findings #3, #4, #5, #6 — all small, localized edits.

ERROR COPY (#3): The error Card currently renders `Impossible de charger le tableau de bord : {error}` where `{error}` is the raw `Error.message`. Replace the visible text with human, actionable French guidance modelled on the component's own good empty-state microcopy (`Aucun contenu ne nécessite votre attention.`) — e.g. tell the editor the dashboard could not load, to retry in a moment, and to contact the developer if it persists. Keep the raw technical `error` string only inside a collapsed/dev detail (a native `<details>`/`<summary>` disclosure whose summary is plain French and whose body shows the raw message) so the developer can still see it without exposing it as the primary message. Do not delete the raw message entirely.

PREPARATION EXCLUSION (#4): `preparation` is a deliberate, first-class radio state in `sanity/schemas/gallery.ts` ("« En préparation » reste dans Sanity"), not an error. Today the `attention` eligibility filter surfaces any `publicationStatus === 'preparation'` document in "À faire maintenant", giving deliberate drafts the same urgency as genuinely broken content. Change the eligibility filter so a document is NOT surfaced when `preparation` (or the legacy `!publicationStatus && isVisible === false` "en préparation" equivalent) is its ONLY reason — i.e. remove the standalone `preparation` inclusion clause from the `attention` filter so an item still appears if and only if it has a real reason (missing required fields → `!requiredComplete`, missing recommended → `!recommendedComplete`, or unpublished draft edits → `hasDraft`). A `preparation` collection that is otherwise complete drops out of the urgent list; a `preparation` collection with missing required fields still appears (correctly, under "À corriger"). Keep `buildAttentionGroups`, `attentionPriority`, and `editorialStatus` internally consistent — the "À finaliser" bucket and the "En préparation" badge logic stay, they just no longer receive preparation-only items pushed in solely by the filter. Do not build a persistent dismiss/snooze store (out of scope for this quick task).

HEADING SEMANTICS (#6): In `AttentionSection`, the group title currently renders as `<Text size={1} weight="bold">{group.title}</Text>`, which Sanity UI's `Text` renders as a non-heading element — so screen-reader heading navigation sees h1 → h2 → h2 and cannot jump between priority buckets. The two real section headings already correctly use `<Heading as="h2">`. Give the four group titles explicit level-3 heading semantics: render the title as a heading (e.g. `Heading as="h3"` styled to match the current size/weight, or add `role="heading" aria-level={3}` to the existing Text). One change to the shared `AttentionSection` covers all four groups. Do not change `MetricCard` (its `Heading` correctly has no `as` prop and must stay a non-heading KPI number — see audit §Accessibility).

DEAD CODE (#5): (a) In the `IntentButton` "Nouvelle collection" JSX, remove the `editorial-dashboard__primary-action` class from the className (it has zero matching CSS rule; keep `editorial-dashboard__header-control`). (b) In `deployment.ts`, `deploymentLabel()` returns an unused hex `color` field that `DeploymentStatus` never reads (it recomputes a Sanity `tone` via a parallel ternary). Consolidate to a single source of truth using the Sanity tone system already used everywhere else: change `deploymentLabel()` to return `{label, tone}` (tone values mapping the existing branches: no run → `default`, status not completed → `caution`, conclusion success → `positive`, otherwise → `critical`), delete the hex `color` field, and update `DeploymentStatus` to consume `status.tone`, deleting its duplicate inline tone ternary. Verify no hardcoded hex colors remain in `deployment.ts`.
